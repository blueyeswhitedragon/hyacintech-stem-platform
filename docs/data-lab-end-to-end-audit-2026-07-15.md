# Data Lab 端到端质量与产品审计

> 审计日期：2026-07-15
> 审计范围：数据种子、Dataset v3 计划、DeepSeek V4P 蒸馏、Student/Tutor/Evaluator rollout、自动门禁、人工标注、工作量审核、匿名仲裁、数据发布、训练资格、双盲评测、灰度部署与 Data Lab 管理界面。
> 核心问题：当前系统能较好地证明“JSON、状态机和血缘存在”，但不能证明“这是一段自然、真实、值得训练的教学对话”。

## 1. 总结结论

本次审计确认，用户指出的三个问题都成立，而且不是局部文案问题：

1. **蒸馏数据的低质量来自上游机制，不适合依靠下游人工逐句清洗挽救。** 当前计划生成器把内部分类名称直接写成学生台词；Student Simulator 被提供预定答案；P2 甚至丢弃模拟器的自然表达并替换成固定事实句；Tutor、Student、Evaluator 默认又是同一个 DeepSeek V4P。结果是结构合规但双方都像“执行脚本的代理人”。
2. **当前实际是五种具体风格加一个 `auto` 入口。风格已经扩散到 63 个代码/脚本/Schema 文件，并进入校准矩阵、标注配额、训练导出、评测和部署门禁。** 但风格真实性主要依赖关键词和问号数量判断，既没有可靠教学收益证据，也明显放大了数据生产和评测工作量。建议停止把风格作为训练、标注和部署的一等维度。
3. **Data Lab UI 缺少“我现在为什么要做这一步、做完会影响什么、有哪些阻断项”的解释。** 更严重的是，页面会把结构门禁的绿色状态写成“自动检查通过”，容易让用户误以为内容质量已经通过；标注员无法修改低质量学生消息，也没有“源数据不可清洗，退回重生成”的入口。

除上述问题外，还发现以下高风险漏洞：

- 同模型自生成的 `stage_contract_rollout` 即使完全未做实质人工修订，也可能获得 SFT 资格；
- `needs_review` 不在活动创建页面的默认候选等级中，且服务端没有强制其双人复核；
- 当前示例中的明显伪人对话，在 import、submit、release 三种校验模式下均返回 `ok`；
- 同一个 `stage-contract-v2` 名称覆盖了多轮不兼容的规则变更，版本号不能证明生成规则一致；
- 冻结发布只阻断 `error`，不要求对 warning 逐项确认，也不要求仲裁理由非空；
- 双盲部署门禁没有最小样本量、置信度、关键安全维度零容忍或 10%→30%→100% 之间的线上观察窗口；
- Gold/Silver 权重只写入数据库和 manifest，没有作用于实际 training JSON，界面容易造成“训练会自动加权”的误解；
- 合成数据没有跨批次内容级近重复门禁，改 ID 或重排 JSON 就可能重复进入训练。

**总体建议：暂停扩量和训练发布，先重建“真实对话质量”门禁和种子生成机制。当前候选应作为回归/反例/生成器调试样本，而不是继续投入大量人工润色。**

## 2. 对用户示例的复盘

示例位于：

```text
data/sft/v3/runs/calibration-30-h0-r3/candidates.json
```

其第一轮为：

```text
学生：我对“高概念降级型-火星基地植物”感兴趣，但还不知道怎样变成课堂里能研究的问题。
导师：你对“火星基地植物”最想保留的那个真实机制或困难是什么？
提示：想想看……
提示：可以是某个特殊条件（如光照、大气、土壤）……
```

### 2.1 为什么学生像假的

`build-dataset-v3-plan.ts` 的 P1 开场直接使用：

```ts
return `我对“${record.scenario}”感兴趣，但还不知道怎样变成课堂里能研究的问题。`;
```

而 `record.scenario` 是内部数据分类：

```text
高概念降级型-火星基地植物
过度宏大型-生态瓶平衡
工程项目型-油污分离装置
```

“高概念降级型”是研究人员的失败模式标签，不是学生会说的主题。内部 taxonomy 被直接泄漏到学生可见文本，是该例伪人感的第一直接原因。

### 2.2 为什么学生会顺着预定答案走

P1 的 `studentVisible.decisionFacts` 已经包含目标方向。火星主题被预先映射为：

```text
每日人工光照时长 → 第7天幼苗株高
```

Student Simulator 能看到 `studentVisible`，因此虽然 Tutor 看不到最终答案，模拟学生仍被脚本驱动到预定的“光照时长”答案。这不是开放探究，只是把答案从 Tutor 泄漏改成由 Student Simulator 主动说出。

### 2.3 为什么导师也像假的

P1 Prompt 同时要求：

- 每轮只能推进一个任务；
- 不给任何示例；
- 不出现候选机制；
- 不进行变量操作化；
- 体现指定风格；
- 输出严格 JSON；
- hints 不能给答案；
- 最终确认使用固定结构和近似固定台词。

模型优化目标几乎全部是“不要违规”，自然承接、真实课堂语气、学生年龄适配没有可执行验收维度。结果容易退化为抽象追问：“你最想保留的真实机制或困难是什么？”

同时，该回复的 hints 实际又列出了“光照、大气、土壤”，构成隐藏候选菜单。当前正则只识别若干“比如 A、B 还是 C”形式，没有覆盖括号中的顿号列表，因此漏检。

### 2.4 当前门禁确实会显示绿色

使用当前代码对该完整记录执行：

```text
validateShareGPTRecord(record, 'import')  → ok
validateShareGPTRecord(record, 'submit')  → ok
validateShareGPTRecord(record, 'release') → ok
```

因此页面上的“自动检查通过”不是历史缓存造成的假象；当前规则确实认为它完全合格。

### 2.5 Evaluator 也没有发现问题

该记录的 Evaluator：

- 与 Tutor 使用相同 provider/model：DeepSeek V4P；
- `evaluatorIndependent=false`；
- 仍给五项全部 2 分并接受；
- 评价中声称“没有隐藏选项或候选答案泄露”。

Evaluator 的评分维度只有阶段纪律、grounding、学生主体性、结构、风格，没有单独的学生真实性、对话自然度、模板感、上下文承接或课堂可用性维度。

## 3. 数据种子与话题质量

### 3.1 Topic Library 不是合格的学生选题库

`data/topic-library.json` 有 120 条记录。抽样和统计结果：

- 69/120 被简单规则识别为泛化教学资源标题或占位式主题；
- 只有 7 种不同的 `independentVariable` 文本；
- 68 条的自变量都是“一个可人为改变的条件”；
- 大量题目来自课程资源标题，而非学生真实兴趣，例如：
  - “实验探究”；
  - “在科学探究中怎样设计实验”；
  - “控制变量法在初中化学实验探究中的应用”；
  - “初中化学实验设计与评价（中考专题复习）”。

这类标题适合检索教学资源，不适合直接变成学生研究兴趣。

当前生产 P1 已停止注入 Topic Library，这是正确方向；但旧 489 数据和 Dataset v3 的种子仍间接受其影响，因此污染还没有从新数据链路中消失。

### 3.2 Dataset v3 名义上只复用“场景”，实际仍读取旧答案

计划声明：

```text
sourceUsage: SCENARIO_SEEDS_ONLY
```

但 `resolveDomainSpec()` 会读取旧记录的：

- `expectedTransformation.independentVariable`；
- `expectedTransformation.dependentDirection`；
- `expectedTransformation.classroomProxy`。

因此并非只复用场景名称，而是继续利用旧数据中的预定转化答案选择领域模板。

### 3.3 领域模板过宽，并存在危险的默认回退

当前只有约 18 个 `DOMAIN_TEMPLATES`。若任何主题都不匹配，则默认使用最后一个模板：

```text
泡腾片颗粒大小 → 完全溶解时间
```

这会把未知主题静默转成泡腾片实验，而不是拒绝生成。

现有 30 格计划中可见多处机械映射：

- “控制变量法在初中化学实验探究中的应用” → 泡腾片颗粒大小；
- “初中化学实验设计与评价” → 泡腾片颗粒大小；
- “工程项目型-油污分离装置” → 过滤材料与浊度；
- “变量混乱型-泥沙沉降” → 过滤材料与浊度；
- “过度宏大型-新能源车续航” → 橡皮筋小车。

部分代理可以作为人工设计起点，但不能依靠关键词模板静默决定，更不能自动视为高质量正向答案。

### 3.4 场景标签、学生画像和可见主题没有分层

当前 `scenario` 同时承担：

- 内部失败模式分类；
- 页面标题；
- Student Simulator 输入；
- 学生开场文本来源；
- 语义守卫 persona 匹配。

应拆为：

```text
internalArchetype   内部风险/失败模式，不对学生显示
displayTopic        学生可理解的主题名称
studentOpening      真实口语开场
privateIntent       模拟学生私有意图，不给 Tutor
validDirections     多个可接受方向，不是唯一答案
curriculumAnchor    课程知识与年级约束
```

## 4. Student Simulator 审计

### 4.1 “简短自然”不是学生画像

Student Simulator 的公共 Prompt 主要是：

```text
你只扮演一名初中生。
简短自然地只回答导师本轮问题。
```

它没有稳定描述：

- 年级、语言能力与词汇水平；
- 是否自信、犹豫、误解、跑题；
- 是否会用不完整句、口语或生活经验；
- 对当前科学概念已掌握到什么程度；
- 哪些错误是合理的学生错误；
- 是否会拒绝 Tutor 的不合理建议。

因此所有学生都高度配合、准确、会说研究术语，并且总能按脚本推进。

### 4.2 P2 直接丢弃模拟器自然表达

P2 要求 Student Simulator 返回：

```json
{"message":"简短自然的学生表达","confirmedFactId":"fact_1"}
```

但代码随后不使用 `message`，而是返回该 fact id 对应的固定原文：

```ts
return { message: fact, confirmedFactId: factId };
```

所以 P2 学生消息必然成为：

```text
我选择白醋稀释液pH，具体水平为……
我要测量……单位……
我会控制……
我确认的步骤是……
每个水平重复3次；安全要求是……
```

这不是模型偶尔不自然，而是代码确定性制造的不自然。

### 4.3 同一模型扮演三个角色

默认 Tutor、Student、Evaluator 都读取同一套环境模型。当前校准记录明确显示三者都是 DeepSeek V4P。

即使输入隔离正确，同模型仍共享：

- 相同语言节奏；
- 相同格式偏好；
- 相同推理盲点；
- 相同“看起来正确”的表达模板。

Evaluator 很容易认可自己同类模型的产出，无法提供真正独立的质量信号。

### 4.4 Thinking 模式下采样策略不可控

Provider 在 DeepSeek thinking 开启时不会发送 `temperature`。因此脚本不能明确控制：

- 学生表达多样性；
- rollout 可复现性；
- Tutor 的措辞探索范围。

运行 manifest 记录了 token 与 thinking 信息，但没有记录一个明确的采样温度策略。

### 4.5 系统触发在 Data Lab 被错误标为学生

P2/P3/P4/P5 的首轮常是：

```text
系统触发：学生已确认选题……
系统触发：学生首次进入过程执行阶段……
系统触发：学生已完成数据收集……
```

模型 API 层可以使用 user role 传递内部触发，但 Data Lab 页面统一标成“学生 · 只读”。这会误导标注员判断自然度，也把系统事件和真实学生语言混在同一种数据角色里。

UI 应显示为“系统触发 · 只读”，训练记录则保留真实模型可见角色和独立的 trigger metadata。

## 5. Tutor Prompt 审计

### 5.1 Prompt 过度强调禁止项，缺少自然课堂示范

当前生产 Prompt 长度约为：

| 阶段 | 字符数 | 行数 |
|---|---:|---:|
| P1 | 3028 | 117 |
| P2 | 3356 | 103 |
| P3 | 2406 | 84 |
| P4 | 2490 | 84 |
| P5 | 2294 | 69 |
| P6 | 1953 | 66 |

长度本身不是错误，但同一行为约束经常在以下位置重复：

- phase prompt；
- pacing guidance；
- stage contract instruction；
- style instruction；
- 动态阶段补充。

模型被训练成优先避免规则码，而不是像真实教师一样回应学生。

### 5.2 固定输出示例被模型直接复制

P1 最终回复在 Prompt 中写了几乎完整的固定台词：

```text
请查看探究问题确认书。下一阶段再正式确定变量、测量方式和实验方案。
```

P5 也有固定的“报告框架已生成”表达。小数据集下，这些句子会大量重复并进入 SFT 目标。

应给“意图和必要信息”，不应给可直接复制的完整 dialogue。

### 5.3 过度禁止支架，反而制造空洞苏格拉底问句

P1 禁止所有示例、候选机制和具体方向，容易逼出：

```text
你最想保留的真实机制或困难是什么？
你想从哪个具体特点入手？
```

对已经困惑的初中生，真实教师通常会使用分级支架，而不是无限开放提问。建议改成状态化支架阶梯：

1. 首轮开放询问；
2. 学生卡住时改写问题；
3. 再卡住时给“思考维度”，不直接给课题答案；
4. 只有明确请求帮助时才给一个类比示例或有限候选，并要求学生解释选择理由。

### 5.4 P2 要求所有数字都必须先由学生发明，不符合真实教学

当前规则把 Tutor 提出的任何新数字都视为硬错误。这会迫使 Student Simulator 先背出完整水平、单位和重复次数。

真实教学中，Tutor 可以：

- 提醒一个合理量级范围；
- 解释为什么 1 次重复不可靠；
- 提议可用单位并让学生确认；
- 在安全和资源限制下给出可行边界。

正确做法不是“Tutor 永远不能建议”，而是给建议建立 provenance：

```text
student_proposed
tutor_suggested_student_confirmed
teacher_required_for_safety
server_derived
```

## 6. 现有数据抽样与量化结果

### 6.1 `calibration-30/candidates.json`

实际只有 25 条候选，而不是完整 30 格：

- P1：3 条；
- P2：2 条；
- P3–P6：各 5 条。

共 49 个 human turn、49 个 Tutor turn。简单统计发现：

- 25 个 human turn 含内部分类名称或“系统触发”等非自然学生内容；
- 13 个 human turn 是完全重复；
- 8 个 Tutor dialogue 是完全重复；
- 当前 `submit` 规则重新检查后：
  - 5 条 `ok`；
  - 7 条 `warning`；
  - 13 条 `error`。

说明候选文件中的历史“已接受”状态不能代表当前合同下仍可用。

### 6.2 典型低质量模式

- 学生看到并使用 `result_a`、`level_1_result` 等内部字段键；
- P2 学生按清单逐项报出完整事实；
- P4 Tutor 先发现异常，再要求学生确认；
- P5 使用“较低/中等/较高”通用模板填充不具体的化学主题；
- P6 把教学资源标题当作“这次实验”的名称；
- Tutor dialogue 和 hints 重复同一问题；
- hints 用括号或顿号给出隐藏选择菜单；
- “我已读取数据”“我已根据真实阶段数据生成”出现高度模板化。

### 6.3 旧 489 条

使用当前 `submit` 规则重新检查：

- 0 条 `ok`；
- 58 条 `warning`；
- 431 条 `error`。

旧 489 已被隔离是正确的。但 Dataset v3 仍从它们读取 scenario、expectedTransformation 和 persona 分类，因此场景层污染尚未完全切断。

### 6.4 训练导出的模板放大效应

`toTrainingShareGPTRecords()` 会把一条多轮对话拆成每个 Tutor turn 一个 SFT 目标。该做法可以匹配在线历史格式，但会重复前序上下文。

当原始对话高度模板化时，重复拆分会放大：

- 固定开场；
- 内部系统触发；
- 固定确认台词；
- 统一学生句式；
- schema key 和流程术语。

因此在小数据集阶段，必须先做近重复与模板占比控制。

## 7. 风格系统审计

### 7.1 当前不是六个具体风格

代码定义五种具体风格：

- `socratic_concise`；
- `warm_companion`；
- `engineering_mentor`；
- `evidence_analyst`；
- `classroom_coach`。

教师 UI 另有 `auto`，所以用户会看到六个选择，但 `auto` 只是稳定分桶到五种风格之一。

### 7.2 风格真实性判断过于表面

当前风格判定主要是：

- 苏格拉底：恰好一个问号且 dialogue ≤180 字；
- 温和陪伴：包含“我理解、你已经、可以先”等词；
- 工程导师：包含“约束、参数、验证”等词；
- 证据分析：包含“数据、异常、不确定”等词；
- 课堂教练：包含“下一步、完成、检查”等词。

这会鼓励模型插入风格关键词，而不是形成真实、稳定的教学行为。用户示例之所以通过苏格拉底风格检查，只因为短且有一个问号。

### 7.3 风格已造成系统性工作量

风格逻辑出现在约 63 个代码、脚本、Schema 和文档文件中，覆盖：

- 教师发布作业；
- 会话创建和 Prompt；
- GenerationTrace；
- 蒸馏 6×5 校准矩阵；
- 活动配额；
- 标注和仲裁；
- 数据发布；
- 双盲评测；
- 部署门禁。

部署要求五种风格分别有结果，理论上把评测工作放大约五倍。

### 7.4 风格矩阵还存在混淆变量

30 格中每个阶段×风格使用不同 scenario。不同风格不是在同一批场景上比较，因此“风格效果”与“主题难度、阶段、数据模式”相互混杂。

### 7.5 建议

**停止把风格作为训练标签和部署门禁维度。**

替代方案：

- 只保留一个统一的“自然、克制、证据导向、适龄”的导师基线语气；
- 把温和、简洁、证据、工程约束等变成横向质量指标，而非互斥风格；
- 校准矩阵改为“六阶段 × 五类交互风险”，例如：
  - 信息模糊；
  - 一次给全；
  - 存在误解；
  - 高概念/工程主题；
  - 安全或异常数据情境。
- 旧 style 字段暂时只保留为 legacy provenance，不再参与新活动、训练、评测和部署。

## 8. 自动门禁：同时过松和过硬

### 8.1 过松：只证明结构，不证明质量

当前门禁没有可靠检查：

- 学生话语是否像真实初中生；
- internal taxonomy 是否泄漏；
- Tutor 是否模板化；
- dialogue 与 hints 是否重复；
- 是否使用内部 schema key；
- 语气是否自然承接学生；
- 对话是否存在真实分歧、犹豫和修正；
- 同批次/跨批次是否近重复；
- Student/Tutor/Evaluator 是否模型独立；
- evaluator score 是否与当前 schema 一致。

### 8.2 过硬：把平台实现细节当作教学质量

以下规则适合作为平台 artifact 校验，但不应让人工导师文本围着它们转：

- `maxRows` 必须精确等于 200；
- 列 key 必须 snake_case；
- 必须存在 `notes` 列；
- 宽表优先；
- P2 所有单位和数字必须先由学生逐字说出；
- P4 接受一轮必须出现至少两个精确表值；
- P4 Tutor 不能使用学生没说出的派生差值或平均值。

这类规则应由 server-composed artifacts、明确 provenance 和阶段状态解决，而不是通过逼迫学生说出数据库字段来满足。

### 8.3 正则门禁存在双重风险

- **漏检**：括号中的“光照、大气、土壤”没有命中隐藏选择正则；
- **误报**：同义改写、否定句、正常教学数字可能被识别为越界；
- **可规避**：换一种标点或词序即可绕过；
- **难维护**：规则不断增长，但不能获得真正语义理解。

建议只把确定性事实来源、状态机、结构和安全对象作为硬错误；自然度、认知负担、支架质量用独立质量评分和人工复核处理。

### 8.4 合同版本没有随不兼容变更升级

`STAGE_CONTRACT_VERSION` 仍是 `stage-contract-v2`，但同名版本下已经发生多轮规则与严重度调整。结果是：

- 历史文件声明 v2；
- 当前代码也声明 v2；
- 当前重新检查却出现大量新 error。

版本字段已不能证明生成时与审核时使用的是同一合同。应至少记录：

```text
contract semantic version
prompt policy version
validator build hash
rule severity profile version
system prompt sha256
artifact composer version
```

### 8.5 autoCheck 是派生快照，不会自动更新

`DatasetSample.autoCheckJson` 在导入时写入。规则更新后需要手动运行 `refresh-data-lab-checks.ts --apply`。

因此批次列表和原始样本提示可能显示旧状态。页面应显示“检查版本/更新时间”，并在发布前自动重算，而不是依赖人工脚本。

## 9. 训练资格与发布漏洞

### 9.1 合成 rollout 不要求实质人工修订

`evaluateTrainingEligibility()` 对非 `production_trace` 的逻辑是：

```text
sourceKind 是 stage_contract_rollout 或 human_authored
+ 批次 ACTIVE
+ 当前合同版本
+ 工作量审核通过
+ 最终被选择
= SFT_ALLOWED
```

它不检查：

- Tutor/Student/Evaluator 是否同模型；
- evaluator 是否独立；
- 学生消息是否人工编写或自然；
- 是否发生 MATERIAL_CORRECTION/HUMAN_REWRITE；
- 是否完成强制双人复核；
- 是否有未解决 warning；
- 是否通过自然度和去重门禁。

这使“模型自己生成、自己评价、人工点通过但未实质修改”的数据可以进入 SFT。

生产回流数据有“必须实质人工纠正”的保护，但合成数据没有同等保护。

### 9.2 `needs_review` 生命周期没有被技术强制

Dataset v3 在 Tutor 和 Evaluator 不独立时写入：

```text
tier = needs_review
```

但活动 UI 的候选等级只有：

- `gold_candidate`；
- `silver`。

没有 `needs_review` 选项。若用户把两个筛选都取消，服务端会把空筛选解释为“全部”，这才可能选到 `needs_review`；这是隐藏行为。

同时，服务端按 `gold_candidate` 决定固定双标。`needs_review` 被当作普通非 Gold 样本，可在单人或 30% 双标模式下流转，违反“同模型评估最多 needs_review，必须双人人工复核”的文档要求。

### 9.3 warning 不阻断 Human Gold 和冻结

- 标注提交允许 warning；
- 工作量审核允许 warning；
- 仲裁允许选择 warning 版本；
- 仲裁理由可以为空；
- UI 默认最终等级为 `human_gold`；
- 冻结发布只拦截 `error`。

warning 本应由人工判断，不一定要阻断。但系统应要求：

- 对每个 warning 明确“已确认可接受/已修复/不适用”；
- Human Gold 必须填写仲裁理由；
- 未确认 warning 不得冻结。

### 9.4 Gold/Silver 权重目前不作用于 training JSON

`goldWeight` 和 `silverWeight` 只写入 `DatasetReleaseItem.weight` 与 manifest。实际 `sharegpt-*-training.json` 没有权重字段，也没有按权重重复采样。

如果外部训练平台不主动读取 manifest，这两个 UI 输入完全不影响训练。

### 9.5 合成数据缺少内容级跨批次去重

导入只检查：

- 文件 SHA；
- `sourceRecordId` 唯一。

重新格式化文件、修改记录 ID 或轻改场景名即可绕过。生产候选有 shingle 近重复检查，合成数据没有同等机制。

### 9.6 发布前没有最小覆盖要求

冻结发布不要求：

- 六阶段都有样本；
- 每阶段达到最小数量；
- 主题/学科分布；
- 学生行为类型分布；
- 自然度分布；
- 关键安全/异常场景覆盖；
- 去重率和模板占比。

只要存在一条人工审定候选，就可以创建冻结版本。

## 10. 人工标注与仲裁流程问题

### 10.1 学生消息只读，使上游低质量不可修复

标注工作台明确规定“只修改导师回复”。这在真实生产回流中合理，但对于合成 rollout 是致命限制：

- 学生开场含内部标签时不能改；
- 学生句子过度正式时不能改；
- 学生无条件配合时不能改；
- 学生提前知道答案时不能改；
- 系统触发被标为学生时不能改。

人工只能把 Tutor 润色得自然，却保留假的输入分布，训练价值仍然很低。

### 10.2 缺少“不可清洗/退回生成器”动作

标注员当前只能：

- 编辑；
- 保存；
- 提交；
- 声明无需修改。

没有：

- 源对话不可修；
- 学生侧不自然；
- 场景错误；
- 应重新 rollout；
- 仅适合 negative/regression。

因此不适合清洗的样本必须先被提交，再由管理员标无效，浪费工作量。

### 10.3 标注员看不到足够上下文

页面只重点展示：

- scenario；
- 对话；
- 阶段目标；
- 风格说明；
- 自动规则。

但 P3/P4/P5 的人工判断还需要：

- Tutor 实际 system prompt；
- trigger 类型；
- approvedPlan；
- dataSchema/dataRows；
- acceptedAnalysis；
- 每轮 VisibleFacts；
- evaluator 是否独立；
- evaluator reasons；
- server-composed artifact 来源。

这些没有以可读形式展示，标注员只能相信自动检查。

### 10.4 仲裁页面没有展示完整候选内容

当前候选卡主要展示：

- dialogue；
- action；
- phase_complete；
- schema 列数。

没有完整展示：

- hints；
- options；
- stage1 confirmation；
- experiment_plan；
- safety_quiz；
- analysis_progress；
- report_sections；
- 与学生消息交错的上下文。

因此仲裁者可能选择 dialogue 更好看、但结构化字段更差的版本。

### 10.5 后端有 MERGE，前端没有合并编辑器

双标的价值不只是二选一，很多时候 A 的表达更好、B 的结构更完整。服务端支持 `MERGE`，但 UI 只提供 SELECT/RETURN/REJECT，无法使用该能力。

### 10.6 “工作量审核”和“质量审核”仍容易混淆

页面说明二者独立，但管理员逐条审核时只看到简化预览和自动检查，可以直接“通过并计 1 条”。warning 不要求说明。

建议把工作量审核限制为：

- 是否完成全部轮次；
- 是否有实质工作；
- 是否明显敷衍；
- 是否遵守任务要求。

内容质量只由独立复审/仲裁决定，避免管理员在两个页面重复判断但标准不同。

## 11. 管理员 UI 审计

### 11.1 概览页只有数字，没有下一步

当前概览卡不可点击，也不区分：

- 健康数量；
- 阻断数量；
- 需要处理的异常；
- 数据当前位于哪一阶段；
- 下一步应该去哪个页面。

建议改成流水线 Stepper：

```text
批次预检 → 活动 → 标注 → 工作量审核 → 仲裁 → 发布预检 → 冻结 → 训练 → 评测 → 部署
```

每一步显示：可继续、等待、阻断、负责人和主操作按钮。

### 11.2 批次页不能进入样本明细

列表只显示汇总，没有网页入口查看：

- 哪些记录 error/warning；
- 规则码分布；
- 各阶段/来源/tier 分布；
- evaluator 独立性；
- 自然度和重复度；
- manifest 内容；
- 具体样本预览。

虽然服务层有 `batchDetail()`，但没有对应详情页面。

### 11.3 网页导入没有预检和来源类型指导

导入表单仍使用：

- ShareGPT clean；
- 人工修订；
- 外部数据。

没有清晰的 Dataset v3 rollout、production trace、human authored 选项，也没有告诉管理员 sourceType 与 record.meta.sourceKind 的区别。

导入后只显示“导入完成 N 条”，不会要求确认 error/warning 比例。

### 11.4 活动创建默认过于危险

默认行为：

- 所有 ACTIVE 批次全部勾选；
- 所有标注员全部勾选；
- 所有阶段全部勾选；
- Gold/Silver 都勾选；
- 五种风格权重都为 1；
- 双人标注默认开启。

页面在启动前不展示：

- 匹配样本数；
- 预计任务数；
- 各参与者任务量；
- error/warning 数；
- needs_review 数；
- 阶段和来源分布。

`StartCampaignButton` 又是一键启动，没有确认对话框和 dry run。

### 11.5 绿色文案具有误导性

“自动检查通过”容易被理解为内容质量通过。建议统一改为：

```text
结构与来源规则未发现问题
```

并始终显示：

```text
不代表自然度、教学质量或 Human Gold
```

### 11.6 发布冻结缺少确认和预览

冻结按钮一键执行，页面不展示：

- 即将发布的具体记录；
- unresolved warnings；
- SFT_ALLOWED/MONITORING_ONLY/BLOCKED 明细；
- 去重结果；
- 六阶段覆盖；
- sourceKind 分布；
- 预计 training turn 数；
- Gold/Silver 权重是否会被外部训练消费。

### 11.7 高级页面暴露实现细节，缺少业务语言

训练页面要求直接填写：

- 原始 JSON 参数；
- 英文状态；
- 外部 task ID；
- 模型 tag。

评测导入只提供一个多文件上传框，不预览文件之间的对应关系。模型页面显示原始状态名和门禁结果，却不解释失败原因和缺少哪些证据。

### 11.8 导航与文档漂移

- 当前导航没有 active 状态和面包屑；
- “高级管理”折叠后，新用户不理解为何发布、训练、模型都藏在里面；
- 管理员指南仍写“没有密码重置、删除账号、无法阻止无需修改同时编辑”等旧限制，但代码已经实现部分能力；
- 文档与 UI 不一致会进一步降低信任。

### 11.9 缺少审计日志页面

系统保存 `DataLabAuditLog`，但管理员没有页面按用户、实体和时间查看关键操作。对于冻结、训练、评测导入和部署，这是必要的可解释性入口。

## 12. 评测与部署门禁漏洞

### 12.1 五风格门禁本身应移除

当前部署要求五种风格分别非退化。由于风格不是可靠的独立能力维度，该要求增加成本，却不能保证真实教学质量。

### 12.2 没有最小样本量或统计要求

每种风格只要：

- 有至少一次 run；
- 至少一个 decisive verdict；
- 胜率 ≥50%；

即可通过该风格门禁。没有最低场景数、置信区间或重复实验要求。

### 12.3 不检查关键维度零容忍

Blind eval 文件有安全、结构、学生主体性等维度，但部署门禁只使用 scenario 层 A/B/tie/inconsistent 汇总。

模型可能整体获胜，但在安全或事实 grounding 上出现严重失败，当前 gate 不会单独阻断。

### 12.4 评测导入验证不足

网页导入主要检查：

- JSON 可解析；
- 有 schemaVersion；
- styleFamily 合法。

没有完整验证：

- transcript A/B 与 verdict A/B 是否一一对应；
- scenario/turn ID 是否一致；
- verdict 是否覆盖全部 transcript；
- 是否存在重复或缺失；
- judge 是否与候选模型独立；
- artifact 是否由可信脚本产生。

管理员误传或手工修改 summary 都可能影响门禁。

### 12.5 10%→30%→100% 没有观察窗口

平台限制了比例顺序，但没有要求每一步之间：

- 至少运行多久；
- 至少积累多少会话；
- 线上结构错误率；
- 安全拦截率；
- 用户退出率；
- 教师驳回率；
- 回滚阈值。

管理员可以连续点击三次，名义灰度并不等于实际灰度治理。

### 12.6 候选作为模型 A 时可能不刷新 gate

`importEvaluation()` 只在 `modelBVersion` 存在时调用 `refreshModelDeploymentGate()`。若候选模型位于 A，导入后不会自动刷新其状态。

## 13. 建议的新质量架构

### 13.1 将门禁分为四层

#### 第一层：确定性硬门禁

只处理能够确定证明的错误：

- JSON/Schema；
- 状态机触发；
- 无来源数字和事实；
- 数据引用不存在；
- 安全黑名单和危险对象；
- server-composed artifact 一致性；
- 上下文缺失；
- 重复 ID/内容指纹。

#### 第二层：教学行为门禁

使用可解释规则和少量独立 judge：

- 是否越阶段；
- 是否替学生完成核心判断；
- 是否一次推进太多任务；
- 是否尊重异常与不确定性；
- 是否与学生当前困难对应。

通常输出 warning，需要人工确认，不应无限扩张正则硬拒。

#### 第三层：自然度与语料质量门禁

新增 corpus-level 检查：

- 内部 taxonomy 泄漏；
- 系统触发误标为学生；
- 学生话语口语度和年龄适配；
- 近重复与模板占比；
- dialogue/hints 重复率；
- Tutor 复述学生的比例；
- 内部 schema key 可见率；
- 学生无条件配合率；
- 每阶段轮数和 dialogue act 分布；
- 主题、学科、行为风险覆盖。

#### 第四层：人工质量裁决

必须明确审核：

- 学生是否真实；
- Tutor 是否自然；
- 对话是否值得训练；
- 是否应该修、重生成、转 negative 或删除；
- warning 是否逐项关闭。

### 13.2 使用单一导师策略，不再做互斥风格

统一目标：

```text
自然、简洁、尊重学生、证据导向、适龄、不过度代劳。
```

不同阶段自然体现不同侧重点，而不是为同一句话强行加“工程”“陪伴”或“证据”标签。

### 13.3 重建种子格式

建议新的种子由教师或课程人员审定：

```json
{
  "displayTopic": "火星基地里怎样让植物获得合适光照",
  "studentOpening": "我想研究火星基地里的植物怎么活下来，可是在学校里没法真的模拟火星。这个还能做实验吗？",
  "grade": 7,
  "curriculumConcepts": ["植物生长条件", "控制变量"],
  "internalArchetype": "high_concept_proxy",
  "privateStudentIntent": "希望保留人工控制环境这一机制",
  "acceptableDirections": ["光照时长", "光照强弱", "水分供给"],
  "unsafeDirections": [],
  "teacherNotes": "不要求唯一映射到光照时长"
}
```

重点是存在多个合理方向，而不是一个 hidden expected answer。

### 13.4 Student side 优先使用人写或真实脱敏语料

优先级建议：

1. 已授权、脱敏的真实学生消息；
2. 教师人工编写的学生回合；
3. 不同模型生成后由人审核的学生回合；
4. 同模型自演 Student/Tutor 只用于压力测试，不作为默认正向 SFT。

### 13.5 为合成数据增加保守训练资格

建议：

- `stage_contract_rollout` 默认 `MONITORING_ONLY`；
- 只有学生侧通过人工自然度审核，且 Tutor 侧完成实质人工修订或双人逐轮确认，才可 SFT；
- 同模型 Evaluator 永远不能提升资格；
- 未解决 warning、重复过高、内部标签泄漏直接阻断；
- Human Gold 必须有非空仲裁理由和 warning closure 记录。

### 13.6 部署门禁改为阶段和风险覆盖

建议维度：

- P1–P6 各自最小场景数；
- 安全高风险集；
- 无来源事实/幻觉集；
- 学生主体性集；
- 异常数据与不确定性集；
- 自然表达人工偏好集；
- 结构成功率和 retry 率；
- 线上教师驳回/会话中断指标。

Critical safety/grounding failure 应零容忍；总体胜率不能抵消关键失败。

## 14. 分阶段整改建议

### P0：立即止损

1. 暂停从现有合成候选创建新的训练发布。
2. 把 `calibration-30` 和 H0 runs 标记为 generator regression/needs redesign，不宣称 Gold。
3. 在 UI 中把“自动检查通过”改为“结构与来源规则未发现问题”。
4. 暂停五风格校准、配额和分风格评测工作。
5. 禁止 `stage_contract_rollout + NO_CHANGE/LIGHT_EDIT` 直接获得 SFT 资格。
6. `needs_review` 服务端强制双人审核和仲裁。
7. 冻结发布前自动重跑当前 validator，并阻断未确认 warning。

### P1：修复生成上游

1. 拆分 internalArchetype 与 student-visible topic。
2. 停止用旧 `scenario` 直接生成学生开场。
3. 停止 P2 用 fixed fact 原文覆盖 Student Simulator message。
4. 领域模板无匹配时拒绝任务，不允许默认泡腾片。
5. Student、Tutor、Evaluator 至少使用不同模型或不同供应商；Evaluator 增加自然度维度。
6. 移除 Prompt 中可直接复制的固定 dialogue。
7. 建立教师审定的小规模种子库。

### P2：重做人工工作台

1. 新增“不可清洗/重新生成/转负例”操作。
2. 区分“系统触发”和“学生消息”。
3. 显示 VisibleFacts、方案、数据表、Prompt、Evaluator 独立性。
4. 仲裁按对话顺序交错显示学生与 Tutor，并展示全部结构字段。
5. 实现 MERGE 编辑器。
6. Human Gold 默认不选；理由必填；warning 必须逐项确认。

### P3：重做管理员体验

1. 首页改流水线 Stepper 和阻断清单。
2. 增加批次详情、样本预览、规则分布、重复率和自然度报告。
3. 活动启动前显示 dry run 和预计工作量。
4. 发布冻结前显示完整 preflight 和二次确认。
5. 训练参数改为结构化表单；高级 JSON 仅作为折叠选项。
6. 模型页显示 gate failures、缺少的评测和线上观察状态。
7. 增加审计日志页。

### P4：重新校准后再扩量

1. 先做六阶段各 3–5 条，共 18–30 条；
2. 学生侧和 Tutor 侧都做人工自然度验收；
3. 两名教育人员独立审核；
4. 用新门禁复验；
5. 通过后再扩到 100；
6. 100 条真实抽检稳定后再考虑 400+。

## 15. 建议的验收指标

新流程在扩量前至少满足：

- 内部 taxonomy 出现在学生可见消息：0；
- “系统触发”被标为学生：0；
- 学生消息含 `result_a/level_1_result` 等内部 key：0；
- exact Tutor dialogue 重复率：<2%；
- 高相似模板回复率：<10%；
- dialogue 与 hints 高重合率：<10%；
- 同模型同时担任 Tutor/Evaluator 的 Gold：0；
- 合成 NO_CHANGE 直接 SFT：0；
- unresolved warning 进入冻结版本：0；
- 每条 Human Gold 均有非空仲裁理由；
- 六阶段各有最小覆盖；
- 关键 safety/grounding error：0；
- 双人自然度偏好通过率达到预设阈值后才扩量；
- 灰度晋级必须满足最小线上会话数和观察时间。

## 16. 最终判断

当前 Data Lab 的血缘、权限、不可变轨迹、授权、脱敏、活动归档和基本状态机设计有明显价值，不应推倒重来。

真正需要重构的是三层：

1. **数据从哪里来**：不能再以旧合成答案和内部分类名称为新正向数据种子；
2. **什么叫质量通过**：不能再把结构合规、风格关键词和同模型评价当成教学质量；
3. **人如何做决定**：标注者和管理员需要看到完整上下文，并能快速拒绝不可清洗的源数据，而不是被迫逐句润色。

用户提出“对这么低的数据做人工清洗失去意义”是正确判断。当前最优策略不是继续清洗，而是先暂停扩量、删除风格维度、重做学生种子与自然度门禁，再让人工审核投入到真正有修订价值的数据上。
