# Data Lab 团队内部速查

本文面向团队内部的管理员和复审人员，重点说明日常操作。标注员请使用单独的录屏教程。

## 一、启动平台

双击项目根目录的 `启动平台.cmd`：

- 选 `2. 本机运行`：需要只在本机使用时启动完整 Next.js 平台，默认地址为 `http://127.0.0.1:3000`。
- 选 `3. 公网审核（ngrok）`：先启动或复用本机平台，再建立 ngrok HTTPS 公网地址，供外部标注员访问。
- 选 `4. 查看状态`：查看本机服务和当前 ngrok 地址。
- 选 `5. 备份数据库`：备份 SQLite 数据库。
- 选 `6. 停止服务`：停止本启动器负责启动的 Next.js/ngrok 进程。

首次在一台新电脑部署时选 `1. 首次初始化`。它会创建 `.env`、安装缺失依赖，并在没有数据库时迁移和初始化数据。若已经存在 `prisma/dev.db`，为了避免误改真实数据，启动器不会自动执行新迁移；升级代码后应先备份，再由维护人员执行 `npm run db:migrate`。

这个启动器覆盖同一个 Next.js 应用里的首页、教学端和整个 Data Lab（账号、任务、标注、审核、发布、训练登记、模型血缘、评测登记）。SQLite 不需要单独启动。每次启动会幂等登记 `.env` 当前选择的运行模型和生产基线，但不会读取或保存 API Key。它不负责启动外部大模型服务、Prisma Studio、训练平台任务或双盲评测脚本；公网模式还要求电脑已安装并配置 ngrok。

## 二、创建账号

进入“高级管理 → 后台账号”。

1. 填写用户名、显示名、初始密码和身份，创建账号。
2. 把初始密码私下交给本人，不要放进公共群聊或教程视频。
3. 已有账号可编辑显示名、用户名和身份，也可重置密码或停用。
4. 有历史任务或审核记录的账号不能永久删除，应使用“停用”；没有任何业务记录的空账号才允许删除。

停用账号会立即失去访问权限，也不会进入新的任务分配名单。

## 三、下发标注任务

进入“任务分配 → 创建任务分配”。

1. 填任务名称，例如“七月第一批人工标注”。
2. 选择数据批次；“最多处理多少条样本”填 `0` 表示全部。
3. 勾选参与人员，并为每人填写“最多领取”。填 `0` 表示该参与者不限条数。
4. 选择审核强度。正式批量工作建议选“每条由 2 人独立标注”；规则稳定且希望提高速度时才选单人标注。
5. 一般不需要展开高级选项；只有需要限制阶段、候选等级或回复风格时再调整。
6. 点击“创建任务分配”，然后在下方活动卡片点击“启动”。草稿未启动时，标注员领不到任务。

活动卡片中：

- “有效任务”是已通过工作量审核的任务数；双标样本的两位标注员各算 1 条。
- “已提交任务”包含尚未审核的提交。
- “待工作量审核”必须由管理员处理。
- “待仲裁”表示两个有效版本仍需最终选版。

如果一个试运行或旧活动不再需要，点击活动卡片右上角的“结束并归档”：

1. 先核对弹窗中的未完成任务、正在编辑任务、未提交草稿、已提交任务、待审核/待仲裁和关联发布版本数量。
2. 填写归档原因，例如“试运行结束，正式标注改用新活动”，再确认归档。
3. 系统会取消所有未完成任务并停止继续分发，但保留负责人、草稿、已提交内容、有效工作量、审核、仲裁和数据集版本。
4. 归档活动中的待审核提交仍可“通过并计 1 条”或“标记无效”，但不能再退回给标注员修改；待仲裁记录仍可接受、合并或拒绝现有版本。

只有从未启动且没有任何业务记录的草稿活动可以“永久删除草稿”。已启动活动不要物理删除，否则会破坏工作量统计和数据集血缘。历史归档活动默认折叠在页面下方，不会进入标注员领取队列。

如果标注员说没有任务，先检查活动是否启动、本人是否被选为参与者、个人领取上限是否已用完。若页面提示“双标任务需要其他标注员完成”，这是正常的双盲隔离，不要把同一样本再次发给本人。

## 四、确认有效工作量

进入“有效标注统计”。

1. 在“逐条审核”中展开“查看本次提交内容与自动检查”。绿色表示自动检查无发现；黄色表示必须人工判断的复核项；红色为硬错误，不能计为有效标注。
2. 内容合格时点击“通过并计 1 条”。
3. 可以修正时，填写审核说明后点“退回修改”；进行中活动的任务会回到原标注员。活动已经归档时不能退回。
4. 提交无效、需要别人重做时，填写说明并点“标记无效”；进行中活动的任务重新开放，归档活动的任务保持取消，原提交都不计数。
5. 结算条数只看每个人的“审核通过/有效条数”，不要按已分配或已提交统计。
6. 需要留档时点击“导出逐条明细 CSV”。后台只记录条数，不记录单价或金额。

工作量审核与数据最终选版是两件事：双标的两个合格提交都可以各计 1 条，但数据集中最终只保留仲裁选中的版本。

自动预检、正式提交、工作量审核和匿名仲裁共用同一套服务端规则。黄色允许流转是为了避免用关键词冒充语义理解，不代表默认合格；审核人员需要结合六阶段文本验收标准逐项判断。红色版本的“通过并计 1 条”会被界面和服务端共同阻止。

## 五、数据仲裁

进入“数据仲裁”。系统对不同标注员的版本进行匿名展示。

1. 比较两个版本是否符合当前阶段目标及结构要求。
   - 红色版本不能接受；
   - 黄色版本可以进入选择，但必须在仲裁理由中说明人工判断；
   - 绿色只代表自动检查无发现，不等于 Human Gold。
2. 选择更好的版本，或按页面提供的方式合并/定级。
3. 两个版本都不合格时退回或拒绝，不要为了清空队列勉强选一个。

只有相关提交先通过“有效标注统计”的工作量审核后，双标样本才会进入可仲裁状态。

## 六、下载完全标注好的数据集

先在“任务分配”确认目标活动满足以下条件：

- “有效任务”为总任务数，即 `有效任务 = 全部任务`；
- “待工作量审核”为 `0`；
- “待仲裁”为 `0`；
- “完成样本”为目标样本总数。

然后进入“高级管理 → 数据版本”：

1. 填写唯一版本号，例如 `stem-sft-2026-07-v1`。
2. 选择刚刚确认完成的标注活动。
3. 一般保持 Gold 权重 `1.5`、Silver 权重 `1.0`，点击“创建版本草稿”。
4. 再次核对活动后，点击“冻结发布”。冻结后不可修改。
5. 冻结成功后下载：
   - `training`：在每条记录首部加入模型可见的 system 风格指令；需要训练出可控回复风格时使用这个；
   - `preference`：符合生产回流资格的“人工修订 chosen / 原模型回答 rejected”配对，用于偏好训练；
   - `clean`：Gold 与 Reviewed Silver 合并后的 canonical ShareGPT 记录，用于人工核对、归档或自行编写训练适配器；
   - `gold`：只含人工仲裁后的 Human Gold；
   - `silver`：只含工作量审核通过的 Reviewed Silver；
   - `manifest`：版本、数量、阶段分布、样本来源、风格分布、训练资格、文件哈希和权重清单，用于留档和复现。

历史冻结版本是在风格训练导出上线前生成的，页面会提示“旧版本未生成风格训练导出”。它们仍可下载原有 clean/gold/silver/manifest，但若要使用新版风格训练格式，应从完成审核的活动创建并冻结一个新版本，不要手工伪造旧版本的风格。

重要：当前系统只要已有合格候选就允许冻结，不会替团队判断活动是否已经 100% 完成。因此必须先按上面的四项条件人工确认，不能提前冻结。

## 七、填写训练登记

这里是外部训练平台的留档页，不会自动提交训练，也不保存平台密钥。先进入“高级管理 → 训练登记”选择父模型并运行资格检查，再去主办方或云平台创建任务。提交或运行状态下，父模型必填；系统会重新检查当前授权、候选状态、人工修正和泄漏状态。

示例：

| 字段 | 示例 | 怎么填 |
|---|---|---|
| 任务名称 | 七月第一轮 SFT | 团队内部可识别的名称 |
| 数据版本 | `stem-sft-2026-07-v1` | 只能选择已经冻结的版本 |
| 基础模型 | `Qwen3.5-35B-A3B` | 外部训练实际使用的底模 |
| 主办方任务 ID | `task-20260713-001` | 外部平台返回的任务号；尚未提交可留空 |
| 模型标签 | `qwen-stem-sft-v1` | 后续双盲采集所使用的稳定标签 |
| 状态 | `RUNNING` | `DRAFT` 未提交、`SUBMITTED` 已提交、`RUNNING` 运行中、`SUCCEEDED` 成功、`FAILED` 失败 |
| 训练参数 JSON | `{"learning_rate":0.00002,"epochs":3}` | 必须是合法 JSON；不知道时填 `{}` |
| 备注 | 使用 training 文件；风格规范 style-v1 | 记录实际数据文件、风格规范、特殊配置或失败原因 |

登记前核对数据版本、基础模型和外部任务 ID，避免用同名记录混淆不同轮训练。

## 八、填写双盲结果

双盲采集和裁判由命令行脚本完成，网页只负责保存结果和汇总。维护人员运行：

每次比较必须让两个模型完成同一种目标风格的采集，并把风格写进标签，避免不同风格的文件互相覆盖。例如评测“证据分析型”：

```powershell
npx tsx scripts/blind-eval.ts collect --scope smoke --style evidence_analyst --tag baseline-evidence
npx tsx scripts/blind-eval.ts collect --scope smoke --style evidence_analyst --tag tuned-evidence
npx tsx scripts/blind-eval.ts judge baseline-evidence tuned-evidence --scope smoke
```

脚本产物位于 `data/blind-eval/`，常见文件为：

- `transcript-<模型标签>.json`：某个模型的完整对话采集结果；
- `verdict-<模型A>-vs-<模型B>.json`：匿名裁判结果。

进入“高级管理 → 双盲评测”：

1. 填评测名称，例如 `qwen-stem-v1-vs-baseline-smoke`。
2. 一次选择两个模型对应的 transcript 文件和同一组 verdict 文件；文件必须是脚本原始 JSON，并包含 `schemaVersion`。
3. 点击“导入评测”。
4. 检查页面显示的模型 A/B 标签、目标风格、scope、产物数量，以及 A 胜、B 胜、平局和不一致数量是否与命令行输出一致。
5. 在“分风格累计结果”中分别检查五种风格；不要只凭总体胜率判断模型是否可以替换基线。

脚本会拒绝把不同目标风格或不同风格规范版本的 transcript 放进同一场裁判，也会拒绝在一次网页导入中混入多种风格。不要手工交换 A/B 标签或改写 verdict JSON；网页会按文件内的标签读取。首次只跑 `smoke` 检查流程，确认模型、裁判和产物都正常后再运行完整范围。

## 九、登记模型与查看生成血缘

进入“高级管理 → 模型与血缘”。

- 当前 `.env` 实际使用的模型会在平台启动时自动登记，并建立一条只读生产基线；不要为同一个运行模型重复建标签。
- 外部训练成功后，可以登记一个新模型版本，并选择父模型与“训练登记”，形成父模型 → 冻结数据/训练任务 → 输出模型的关系。
- “完整追踪会话”表示从会话创建起，每一条正式导师回复都有生成轨迹；“历史不可验证会话”不能因为后来补了几条新轨迹就被当成全程可信数据。
- GenerationTrace 保存模型身份、风格、契约结果和内容指纹。它不保存 API Key，也不重复保存学生原始请求正文；只有学生已经明确授权回流时，才额外保存模型当轮实际看到的完整系统上下文，供后续训练记录保持可追溯和不缺上下文。
- “模型与血缘”页面同时提供经过门禁约束的灰度和回滚控制；不能绕过 10% → 30% → 100% 顺序。

如果 `.env` 中的外部模型名称可能指向不断变化的模型，必须设置唯一且稳定的 `LLM_MODEL_TAG`。同一标签一旦绑定 provider/model，就不能改绑到另一个模型。

## 十、处理正式使用中发现的问题

1. 教师发布作业时，只有确实计划收集改进信号，才勾选“允许学生自愿授权脱敏对话用于模型改进”。默认保持关闭。
2. 学生进入作业后可以同意、拒绝或撤回。任何选择都不影响作业进度和成绩，团队成员不得把授权作为完成作业的条件。
3. 学生授权后，教师在审核详情的“提名模型改进候选”中选择具体导师回复并填写问题说明。不要提名整段会话，也不要把提名描述成“已经加入训练”。
4. 管理员进入“高级管理 → 线上候选池”，只查看脱敏快照、脱敏计数和重复检查结果：
   - 内容仍含身份线索、附件内容或没有实际问题时，拒绝；
   - 精确重复会被系统直接阻断；
   - 近重复需要人工判断，不要为了增加数量重复收录；
   - 通过仅表示可以进入人工纠正流程，不表示可以直接训练。
5. 勾选通过的候选并填写批次名，转换为 `production_trace` 隔离数据批次。之后仍必须走任务分配、人工实质修订、工作量审核和仲裁。

学生撤回后，候选会变为 `WITHDRAWN`，不能进入新的标注活动或冻结版本。若数据已经用于完成外部训练，平台只能阻止后续继续使用，无法直接消除已经形成的模型参数影响。

标注生产回流批次时必须选择人工变换类型：“无需修改”和“轻微润色”只用于监测；“实质纠正”表示修复内容、结构、阶段或安全逻辑；“人工重写”表示主要导师回答由人重新构造。服务端会计算真实差异，轻微改字不能冒充人工重写。后台也会阻止任何人审核或仲裁自己写过的版本，不能通过临时修改账号身份绕过。

## 十一、评测门禁、灰度和回滚

1. 新训练模型先在“模型与血缘”登记，关联父模型和成功训练登记。
2. 分别对五种目标风格运行双盲评测；产物中的模型标签必须与注册表完全一致。
3. 门禁只有在训练资格无阻断、总体结果不退化、五种风格各自有有效结果且不退化、不一致比例未超限时才显示 `PASS`。
4. 在“模型与血缘 → 灰度部署控制”依次执行 10%、30%、100%，不能跳级。
5. 平台按会话稳定分桶，正常灰度不会让同一对话中途换模型。
6. 出现结构、安全或效果问题时点击“一键回滚”；紧急回滚会把仍绑定候选模型的会话切回上一安全基线。

部署记录保存门禁快照、上一模型、灰度比例和操作人。不要直接修改 `.env` 冒充灰度；如需紧急禁止部署操作，可设置 `ENABLE_MODEL_DEPLOYMENT=false` 并重启。
