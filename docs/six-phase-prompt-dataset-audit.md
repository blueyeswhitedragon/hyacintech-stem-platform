# 六阶段 Prompt—流程—数据集一致性审计

> 审计对象：当前生产教学流程、六阶段 Prompt、DSV4 蒸馏机制，以及 `data/sft/sharegpt-distill-dsv4-all-clean.json` 中 489 条记录。
>
> 目的：在重构整套数据生产流程前，冻结每个阶段允许和禁止的行为，识别 Prompt、运行时状态机、训练记录和现有数据之间的漂移。

## 1. 结论摘要

现有 489 条数据不只是 P1 局部存在问题，而是生成机制与实际运行流程存在系统性差异：

1. 蒸馏没有复用生产环境的完整阶段 Prompt。
2. `expectedTransformation`、`priorSummary` 和 Persona 素材向生成模型泄露最终答案。
3. 一次调用同时编写学生和导师整段对话，不是真实逐轮 rollout。
4. 生成时可见的阶段 Prompt、数据表和前序摘要，没有完整序列化进最终模型可见的 ShareGPT 消息。
5. P1/P2 固定轮数、P3–P6 固定单轮，与真实状态机不同。
6. 当前训练导出只增加风格 system message，没有加入生产阶段 Prompt 和动态上下文。
7. 校验器主要检查 JSON 和结构字段，几乎不检查教学阶段边界和上下文真实性。
8. 部分数据包含线上请求会被黑名单提前阻断的输入，实际教学中不会进入 LLM。

因此，若本轮准备整体重构，建议将当前 489 条降级为：

- 规则回归集；
- 人工修订来源；
- chosen/rejected 偏好来源；
- 新蒸馏流程的对照基线；

不建议继续将其整体视为可直接训练的正向 SFT clean 数据。

---

## 2. 审计方法

本次审计包括：

- 阅读生产六阶段 Prompt；
- 阅读 `getPromptForPhase()` 的安全、节奏、风格和动态上下文注入；
- 阅读阶段推进、教师审核、前端自动触发和数据持久化逻辑；
- 阅读蒸馏计划、任务构造、Persona、Topic Library 和清洗逻辑；
- 对 489 条记录执行全量结构和语义规则扫描；
- 按阶段、批次、来源执行固定哈希分层抽样；
- 对照训练导出时模型实际可见的 messages。

自动规则用于定位风险，不等价于最终人工裁决；但高比例命中和人工样本共同确认了系统性漂移。

## 3. 全阶段通用合同

### 3.1 通用白名单行为

所有阶段均允许：

- 回应学生刚刚提供的具体信息；
- 每轮聚焦一个核心任务，最多一个轻量追问；
- 给思考方法，不直接替学生完成核心判断；
- 说明当前阶段目标和下一步动作；
- 使用与当前阶段匹配的结构化字段；
- 给与风险成比例的安全提醒；
- 明确区分观察、解释、结论和不确定性；
- 只引用当前 Prompt、实际历史和结构化阶段数据中存在的信息；
- 当证据不足时明确要求补充，而不是编造。

### 3.2 通用黑名单行为

所有阶段均禁止：

- 使用生成器隐藏参考答案替学生决定内容；
- 引用训练 messages 中不可见的数据、材料、步骤或分析；
- 编造数字、日期、设备参数和实验结果；
- 同一轮堆叠多个核心问题；
- 隐藏 ABC 式引导；
- 将下阶段任务提前完成；
- 替学生写结论、反思或核心判断；
- 把相关性表述成确定因果；
- 删除或美化异常数据；
- 产生线上状态机永远不会触发的对话；
- 在结构化字段与动作类型不一致时仍视为 clean；
- 用任务优先级代替输出质量决定候选等级。

## 4. P1 选题定向合同

### 4.1 白名单

- 理解原始兴趣；
- 找到学生希望保留的机制、困难或约束；
- 判断课堂可行性和安全性；
- 引导学生形成粗略课堂代理；
- 形成具体研究问题；
- 确认“拟改变因素方向”和“关注现象方向”；
- 说明代理实验与原主题的关系；
- 输出一次且仅一次选题确认成果。

### 4.2 黑名单

- 正式确定自变量水平或梯度；
- 确定因变量操作定义；
- 追问测量方法或具体指标；
- 逐项列控制变量；
- 确定材料、步骤、分组、重复次数；
- 设计数据表；
- 给计算公式；
- 让学生从多个现成课题、参数或指标中选择；
- 在确认成果后继续留在 P1 生成额外确认轮。

### 4.3 量化结果

P1 共 171 条，全部固定为 3 组 human/gpt：

- 99 条在确认前使用“自变量/因变量”正式术语；
- 约 86–91 条提前讨论测量方式或指标；
- 27 条提前确定水平、梯度或组别；
- 21 条提前讨论控制变量；
- 11 条提前讨论步骤、记录表、重复或计算；
- 保守规则下约 112–114 条至少存在一类 P2 越界，约占 65%–67%；
- 171 条最终确认均输出了非空 dependent；
- 8 条最终确认还包含控制变量；
- 6 条记录出现两次阶段确认成果。

分层人工样本中，14/18 明确进入 P2 内容。

### 4.4 根因

- P1 生成器直接看到最终研究问题、自变量和因变量；
- P1 `priorSummary` 已经是完整选题确认书；
- Persona/Topic 素材直接包含变量答案；
- 蒸馏 Prompt 没有复用生产 P1 边界；
- 固定三轮迫使模型在已完成后继续对话；
- 质量检测只识别少量固定短语，且跳过最终确认回复。

### 4.5 处置

- 当前 P1 不进入新 SFT 正向集；
- 作为 P1 边界回归和 rejected 来源；
- 新流程下全量重新 rollout。

## 5. P2 方案设计合同

### 5.1 白名单

- 正式确定自变量和水平/梯度；
- 确定因变量及测量方式；
- 确定控制变量；
- 确定材料、步骤、重复次数和安全措施；
- 生成与方案一致的数据表；
- 信息不足时一次只追问一个缺口；
- 数据表只在方案已成型后生成；
- 数据表动作使用 `confirmation`。

### 5.2 黑名单

- 重新替学生选择研究主题；
- 在学生未参与时一次性代写完整方案；
- 提前讨论实验结果、趋势或结论；
- 未确定测量方式就生成表格；
- schema 与对话中的变量不一致；
- schema 已生成却使用非 confirmation 动作；
- 同一记录反复生成多份冲突 schema；
- 使用长表结构破坏当前图表组件的直接比较能力；
- 输出重复 key、非法 key、无分析指标或错误行数约束。

### 5.3 量化结果

P2 共 136 条：

- 125 条固定为 4 组 human/gpt，10 条为 3 组，1 条为 2 组；
- 64 条首条学生消息已经一次性给出大部分完整方案，导师主要在验证预设答案；
- 50 条包含较强的直接方案建议或代设参数；
- 29 条包含多次 schema；
- 30 条在最后一轮之前已经输出 schema；
- 29 条 schema 与 action 不一致；
- 1 条使用非法动作 `require_confirm`；
- 136 条最终 schema 的 `minRows` 全部为 1，而生产归一化要求至少 3；
- 38 条使用 group/condition/treatment 长表列，与生产 Prompt 的宽表原则冲突；
- 3 条最终 schema 存在重复 key；
- 9 条存在非 snake_case key；
- 3 条没有数值分析列；
- 约 15 条没有明显安全信息；
- 至少 1 条提前讨论结果或趋势。

### 5.4 产品适配漂移

当前 ChartViewer：

- 第一列作为 x 轴；
- 其余数值列作为多条 y 序列。

因此大量 `day + group + value` 长表无法直接按组绘制多序列，数据设计与 UI 假设不一致。

### 5.5 处置

- P2 应按新结构合同重新生成或深度修订；
- schema 需要共享、机器可执行的规范；
- 不再依赖运行时 `normalizeSchema()` 静默修复训练目标。

## 6. P3 过程执行合同

### 6.1 白名单

- 首次进入时完成与当前实验相关的结构化安全问答；
- 指向实际数据录入面板；
- 提醒真实记录、时间点和异常备注；
- 对操作问题提供不改变核心方案的排查建议；
- 安全风险出现时停止危险操作并提供与原研究问题一致的安全替代；
- 允许记录失败和异常，不删除数据。

### 6.2 黑名单

- 编造数据或观察；
- 提前分析趋势或形成结论；
- 改变研究问题、自变量或核心实验设计；
- 在无教师重新审核时增加新实验条件；
- 给出输入中不存在的具体日期、数值和参数；
- 输出线上黑名单会提前阻断的危险实验指导；
- 只给泛化安全提醒，不说明在哪里记录数据；
- 用一条样本代替首次进入、异常处理和持续记录三个不同场景。

### 6.3 量化结果

P3 共 45 条，全部只有 1 组 human/gpt：

- 0 条输出 `safety_quiz`；
- 25 条没有明确指向数据表/备注/录入面板；
- 4 条建议改变核心方案或增加新条件；
- 2 条提前进入分析；
- 8 条导师回复增加学生输入中没有的数字或参数；
- 至少 1 条导师回复包含生产黑名单危险词或危险替代；
- 至少 1 条学生输入在线上会在调用 LLM 前被黑名单阻断。

### 6.4 流程漂移

生产流程要求：

- 教师批准后进入 P3；
- 首次进入提示包含结构化安全问答；
- 安全问答通过后录入数据；
- 可进行多次异常排查；
- 点击按钮完成数据收集。

现有数据没有覆盖完整状态机，主要是孤立的单轮操作提醒。

### 6.5 处置

P3 需要按场景重新规划并生成：

1. 首次进入 + safety quiz；
2. 正常记录；
3. 异常但保留数据；
4. 安全中止与等价替代；
5. 设备/材料故障；
6. 完成数据收集前检查。

## 7. P4 数据分析合同

### 7.1 白名单

- 3→4 后系统主动开场；
- 明确已读取实际数据；
- 每轮只引导一个分析动作；
- 要求学生引用具体数据；
- 比较组间差异、时间趋势、最大最小、平均或异常；
- 区分观察、解释和结论；
- 保留异常值并讨论影响；
- 证据不足时要求补充或降低结论强度；
- 至少形成两轮有效分析。

### 7.2 黑名单

- 等待学生自己猜测如何开始分析；
- 没有数据仍声称“我看过数据”；
- 引用模型可见输入中不存在的数值；
- 直接说出完整趋势或最终结论；
- 将相关性写成确定因果；
- 一轮连续问多个分析问题；
- 忽略异常和不确定性；
- 使用与初中阶段不匹配的复杂统计；
- 只生成一轮样本，却以此代表需要两轮完成的产品流程。

### 7.3 量化结果

P4 共 65 条，全部只有 1 组 human/gpt：

- 0 条覆盖系统触发的 3→4 主动开场；
- 43 条学生消息没有明确数值；
- 18 条助手引用学生消息中不存在的数字或参数；
- 19 条直接陈述趋势或分析结果；
- 10 条存在因果过度表述风险；
- 16 条没有明确要求学生引用数据证据；
- 12 条一轮提出超过两个问题；
- 至少 3 条直接使用隐藏的通用 A/B/C 数值模板；
- 全部不覆盖产品要求的两轮分析状态。

### 7.4 上下文漂移

生成 P4 时，模型通过隐藏 `priorSummary` 看到了实验数据；但最终 ShareGPT record 通常只保留学生一句概述，没有把完整数据作为 system/user 消息写入训练记录。

训练时模型可能看到：

```text
学生：我记录了三天数据，A组变化不大，B/C上升很快。
```

目标却写：

```text
A组从0到2，B组从1到5，C组从2到6。
```

这是直接训练模型引用不可见数据。

### 7.5 处置

P4 需要重构为：

- transition assistant-only 入口；
- 模型可见的实际 rows/schema；
- 多轮 evidence-first 分析；
- 有效分析进度而非单纯消息计数；
- reactive 与 transition 两类数据分开。

## 8. P5 报告成型合同

### 8.1 白名单

- 使用真实研究问题、方案、数据和分析；
- 自动生成目的、假设、材料、步骤、数据概述和分析框架；
- 只填写有证据支持的内容；
- 缺失内容明确标记并要求学生补充；
- conclusion 和 reflection 保持学生所有权；
- 数据概述引用真实 rows；
- analysis 引用 P4 中学生已经完成的证据判断；
- 首次自动生成使用明确的系统触发语义。

### 8.2 黑名单

- 使用隐藏但训练 messages 中不可见的前序摘要；
- 编造材料、步骤、组别、天数和数值；
- 使用通用 A/B/C 数据填充任意主题；
- 把“待学生补充”当成已经自动预填完成；
- 在 analysis 中直接写最终结论；
- 生成一篇可直接提交的完整报告；
- 声称“已根据你的数据”，但模型可见输入中没有数据；
- 与 P2 schema、P3 rows 或 P4 分析不一致。

### 8.3 量化结果

P5 共 42 条，全部只有 1 组 human/gpt，且全部输出 report_sections：

- 39 条学生消息没有明确数值；
- 37 条 report_sections 出现学生消息中没有的数字或参数；
- 24 条使用“三天/Day1–Day3”通用数据叙述；
- 25 条使用 A/B/C 组模板；
- 16 条使用精确的通用 `0,1,2 / 1,3,4 / 2,5,6` 数据模式；
- 33 条 section 包含“待学生补充/……”等占位内容；
- 12 条 analysis 已经带有结论性判断；
- 4 条明显使用“帮你整理好了/报告草稿”等代写语言；
- 42 条动作全部为 text_input，未区分首次自动生成的 info/transition 语义。

### 8.4 架构根因

生产 `Stage2Data` 只结构化保存数据表 schema，没有结构化保存：

- 自变量与水平；
- 因变量操作定义；
- 控制变量；
- 材料；
- 步骤。

生产 `Stage4Data` 又只保存 `analysisCount`，没有保存：

- 学生观察到的趋势；
- 引用的证据；
- 异常判断；
- 结论强度。

`buildPriorSummary()` 实际只包含：

- P1 确认书；
- P2 表格列；
- P3 rows；

没有结构化 P2 方案，也没有 P4 分析摘要。P5 只能从长聊天历史猜测材料、步骤和分析。

蒸馏则用隐藏的合成摘要补齐这些缺口，最终产生大量训练输入不可见的内容。

### 8.5 处置

P5 应在扩展 Stage2/Stage4 结构化状态后重新生成，不建议在当前上下文模型上修修补补。

## 9. P6 结果反思合同

### 9.1 白名单

- 基于实际报告、教师评价和数据局限提出一个反思问题；
- 引导学生自己识别误差来源；
- 引导学生提出可执行改进；
- 讨论证据能支持到什么程度；
- 迁移应用不超出当前研究机制；
- 最终反思由学生提交。

### 9.2 黑名单

- AI 直接给出完整误差分析和改进方案；
- 同一轮堆叠过程、误差、改进、迁移多个问题；
- 引入新的研究任务或复杂工程设计；
- 提前完成阶段或使用 confirmation；
- 编造教师反馈、实验细节和新数据；
- 将可选聊天行为当成产品主提交路径。

### 9.3 量化结果

P6 共 30 条，全部只有 1 组 human/gpt：

- 9 条 `phase_complete=true`；
- 2 条使用 production Prompt 明确禁止的 confirmation；
- 13 条一轮提出超过两个问题；
- 6 条直接给出原因或改进方案；
- 8 条引入新的实验、迁移或工程扩展；
- 部分回复在询问学生的同时又标记阶段完成。

### 9.4 产品流程漂移

当前产品的核心 P6 路径是：

- 展示报告和教师评价；
- 学生在 Stage6Panel 文本框中直接填写反思；
- POST `stage6-respond` 完成项目。

P6 Prompt 对应的是左侧可选聊天，而不是完成阶段所必需的主流程。当前数据没有明确区分：

- 可选反思辅导；
- 最终学生提交；
- 系统完成动作。

### 9.5 处置

重构前需先确认 P6 产品定义：

- 方案 A：保留表单为主，LLM 仅提供可选辅导；
- 方案 B：先由 LLM 完成一轮证据化反思引导，再允许表单提交。

数据必须与最终选择一致。

## 10. 跨阶段系统性漂移点

### D1. Prompt 双轨

生产 Prompt 与蒸馏 Prompt 独立维护，阶段边界不一致。

### D2. 答案泄露

`expectedTransformation`、Persona 脚本和 priorSummary 同时给出最终答案。

### D3. 隐藏上下文未写入训练 messages

生成时模型看到的数据、摘要和任务要求，最终训练 record 中不可见。

### D4. 训练导出没有生产阶段 Prompt

`toTrainingShareGPTRecord()` 只添加风格 system instruction，没有添加生产 phase Prompt、dataRows、priorSummary 或 transition context。

### D5. 同一模型同时扮演学生和导师

整段对话一次生成，学生行为过于配合，无法模拟真实偏离、犹豫和误解。

### D6. 固定轮数与状态机不一致

- P1 永远三轮；
- P2 主要四轮；
- P3–P6 永远一轮；
- 实际产品按结构化成果、按钮、教师审核和有效分析推进。

### D7. 历史消息格式不一致

蒸馏多轮记录中，之前的 assistant message 是完整 ChatResponse JSON；生产聊天历史持久化并回传的主要是展示用 dialogue 文本。训练上下文与运行时历史形态不同。

### D8. 阶段转场不一致

- P1→P2 使用前端伪用户触发语；
- P2→P3 使用教师审核后固定 assistant 提醒；
- P3→P4 无任何过渡消息；
- P4→P5 使用前端伪用户“开始报告成型”；
- P5→P6 使用教师审核；
- 蒸馏没有准确建模这些转场。

### D9. 结构化阶段状态不足

- P2 缺少正式变量、材料、步骤；
- P4 只有 analysisCount；
- P5 因此无法可靠构造报告；
- P6 缺少结构化反思维度。

### D10. 数据表规范不统一

生产 Prompt、蒸馏清洗、DataLab 校验、运行时 normalizeSchema 和 ChartViewer 对 schema 的假设不同。

### D11. 安全链路不一致

部分数据包含线上会在 LLM 调用前被黑名单拦截的用户输入；P3 又没有训练生产要求的结构化 safety_quiz。

### D12. 校验器偏结构、轻教学

正式 release validator 对 P3–P6 几乎全部判定通过，无法发现数据伪造、越界和上下文缺失。

### D13. 候选等级不是质量等级

`gold_candidate` 主要由 phase、priority、persona difficulty 和 failure mode 决定，不由最终输出质量决定。

### D14. 通用合成数据污染

P4/P5 大量使用固定 A/B/C 三天数据模板，与具体主题、表格 schema 和人类输入不一致。

### D15. 风格标签后置

原始蒸馏内容生成时没有按最终 styleFamily 生成，后续训练导出才添加风格 system message。若人工修订没有真实重写风格，内容与标签可能不匹配。

## 11. 建议的重构目标

### 11.1 单一生产合同

建立共享：

```ts
buildTutorPrompt({ phase, context, triggerType })
validateStageResponse({ phase, triggerType, response, context })
```

生产、蒸馏、盲评和 DataLab 共用。

### 11.2 真实逐轮 rollout

- Student Simulator 与 Tutor Generator 分离；
- Tutor 使用生产 Prompt；
- Student 不看到 expected answer；
- Evaluator 单独看到 expectedTransformation；
- 按产品结构化信号动态停止。

### 11.3 上下文必须模型可见并可追溯

训练 record 必须显式包含生成时使用的：

- phase Prompt 版本；
- style Prompt；
- assignment/topic context；
- stageData 摘要；
- dataRows/schema；
- transition trigger；
- prior analysis evidence。

不能再依赖生成时隐藏、训练时丢失的 Prompt 内容。

### 11.4 扩展阶段状态

建议新增：

- P1：factorDirection、phenomenonDirection；
- P2：independent levels、dependent measurement、controlled variables、materials、procedure、safety plan；
- P4：observations、evidence citations、anomalies、interpretations、analysis progress；
- P6：limitations、improvements、transfer、student reflection status。

### 11.5 转场是一等公民

定义显式 trigger：

```text
USER_MESSAGE
STAGE_ENTER
STAGE_TRANSITION
TEACHER_APPROVAL
REPORT_BOOTSTRAP
OPTIONAL_COACHING
```

不要再用可见伪用户消息模拟系统触发。

### 11.6 新质量门禁

每条数据必须经过：

1. JSON/结构契约；
2. 阶段白名单/黑名单；
3. 上下文可见性检查；
4. 数据引用一致性；
5. 学生主导性；
6. 安全策略；
7. 独立 LLM judge；
8. 人工抽样或高风险全审。

候选等级在生成后根据质量决定，而不是生成前由任务 priority 决定。

## 12. 数据处置建议

既然准备整体重构，建议：

- 489 条全部退出新 SFT clean；
- 按问题标签进入 legacy/quarantine；
- 可修订记录作为人工工作底稿；
- 越界记录用于 rejected preference；
- 高价值情境用于新 rollout 的任务种子；
- 新数据通过六阶段共享合同后再形成 dataset-base-v2。

这不是因为 489 条每条都不可用，而是它们的生成证据无法证明与新生产流程一致。整体重构时，保留其“场景价值”，不保留其“正向训练资格”。

## 13. 重构优先级

### P0：先冻结产品状态机和阶段合同

- 确认 P1 是否保留粗略因素方向；
- 确认 P3→P4 主动过渡；
- 确认 P6 是表单主导还是聊天主导；
- 确认转场和生成追踪模型。

### P1：扩展 StageData 和结构化 ChatResponse

尤其是 P2 方案和 P4 分析证据。

### P2：统一生产 Prompt、蒸馏 Prompt 和校验器

建立单一来源。

### P3：实现逐轮 Student/Tutor/Evaluator rollout

去除答案泄露和固定轮数。

### P4：建立新审计与回归套件

将本报告中的白名单和黑名单变成可执行测试。

### P5：小规模校准后重新生成六阶段数据

先每阶段 10–20 条，人工通过后再扩量。
