# Data Lab 界面与动线重设计方案

> 起草日期：2026-07-16。基于三路审计（管理页文案、标注/复审工作台、管理员动线）与 TopicCard 链路精读。
> 背景约束：训练与推理部署外包给参赛方算力平台（学校后续可能提供同机算力）；DataLab 的定位是**数据生产车间 + 交付出口 + 结果登记簿**，不是训练操作台。

## 0. 问题诊断总纲

三类混乱，根源各不相同：

1. **文案层**：约 60 处内部术语/字段名/英文枚举直接当 UI 文案（familyKey、FinalizedTutorTurn、DRAFT/APPROVED、baselineStructureFailureRate…），8 处 `window.prompt` 收集正式治理信息。这是纯粹的收尾债，机械修复即可。
2. **动线层**：概览页流程卡到"定稿"就断；无任何"当前批次卡在哪一步"的视图；门禁 UI/服务端不一致（生成 180 按钮永远可点）；导航语义错位（部署藏在"历史与后台"折叠区）。
3. **概念层（最深）**：TopicCard 的表单把"给 AI 编译器的输入格式"直接暴露成"给人填的表单"。30+ 个字段平铺，字段名来自代码 schema（探究桥/保留机制/内部 archetype/兼容 subject），填写者看不到任何字段的下游用途——不知道 `levels` 会变成数据表的列、`studentOpening` 会变成 P1 学生开场白。**填不明白不是文案问题，是缺少"所填即所得"的反馈回路。**

## 1. 信息架构重组

按"数据生产流水线 → 数据交付 → 模型迭代登记 → 后台"四层重排导航：

```
概览                    流水线指挥台（按角色显示不同内容）
── 数据生产 ──
  话题库                /data-lab/topics（两个 tab：素材池 | 话题卡）
  案例批次              /data-lab/runs（以 run 为中心，Smoke→Cal→Trial→Full stepper）
  初审工作台            /data-lab/first-review（annotator）
  定稿工作台            /data-lab/final-confirmation（reviewer）
  案例退回处理          /data-lab/case-quality（admin；也可作为案例批次页的一个 tab）
── 数据交付 ──
  数据版本              /data-lab/releases（冻结 → 下载 → 交给算力平台）
── 模型迭代 ──
  模型档案              /data-lab/models（每个模型版本一条时间线：数据 → 训练 → 评测 → 部署）
── 后台 ──
  账号 / 备份 / 历史数据（旧流程只读）
```

要点：
- 删除「评测与模型」这个名不副实的导航项；evaluations 的导入功能并入模型档案时间线的"评测回填"节点。
- 「模型血缘与部署」从折叠区提升为一级导航「模型档案」。
- 每个导航项旁显示该阶段待办计数徽标。

## 2. 概览页 = 流水线指挥台

**按角色渲染完全不同的首屏：**

- **annotator / reviewer**：一张大卡——待办数量 + 一句话职责说明 + 3 步操作指引 +「开始领取」按钮。不显示全管线统计、不显示"旧流程已冻结"横幅、不显示架构自述。
- **admin**：
  1. 顶部一条横向 6 阶段流水线（话题库 → 案例批次 → 双审 → 数据版本 → 训练/评测 → 部署），每阶段状态色：未开始 / 进行中 / 等待他人 / 被门禁阻断 / 完成。
  2. 每阶段下一行"下一个动作"：如「Smoke 6：4/6 已定稿，等待 reviewer 完成 2 条」「Calibration 已通过 → 可创建 Trial 36」，直接链到对应按钮所在页。数据源全部已有（`tutorWorkflowCounts`、三个 qualityReport、model status），无需新模型。
  3. 现有 7 张统计卡改为可点击、直达对应工作页。

删除首页副标题「模型只生成教学语言；状态、阶段推进和结构化产物由 Extractor 与服务器拥有」这类架构自述；引导文案统一为"做什么 / 做完会怎样 / 被什么阻断"三段式。

## 3. TopicCard 重设计（概念层修复，重点）

### 3.1 心智模型先行

页面顶部用一句话+一张示意图定义：

> **一张话题卡 = 一个真实情境 + 2~4 条学生可以走完六阶段的研究路线。**
> AI 会用它批量生成"学生提问 + 数据表 + 导师任务"，你审核的是：情境真不真、路线走不走得通、边界安不安全。

术语全面更名（仅 UI 层，代码字段不动）：

| 代码/现名 | UI 新名 | 说明 |
|---|---|---|
| inquiryBridge 探究桥 | **研究路线** | "学生可能选的一个具体研究问题及其做法" |
| retainedFeature | 保留的真实机制 | 附例句 |
| factor | 要改变的条件（自变量） | |
| phenomenon | 要观察/测量的结果（因变量） | |
| levels | 测试档位 | 说明"会直接变成学生数据表的列" |
| internalArchetype | （隐藏，自动生成） | 高级折叠区可见 |
| subject | 旧版学科分类（兼容） | 折叠进高级区 |
| familyKey | （从 UI 消失） | 见 3.4 |

### 3.2 主路径 = 审核编译产物，不是手工填表

页面重心倒转：素材池「双模型编译」产出的 DRAFT 卡是常规入口，管理员的工作是**审核和修正**；手工创建整卡收进「高级：手工建卡」折叠区作为兜底。审核视图突出 Critic 问题 + 需人工确认的字段，而不是 30 个空输入框。

### 3.3 编辑表单改为四步分节（向导式或锚点分节）

1. **① 情境**：学生可见标题、学生开场白、真实需求、核心机制、活动模式/情境模块（中文选项：科学探究/工程设计/混合；生命健康/能源环境/智能信息/航空航天/深地深海）。
2. **② 研究路线**（≥2 条，就地解释"为什么至少两条：避免学生只有唯一答案可走"）：每条字段带真实示例占位符——
   - 要改变的条件：`如：豆芽每天的光照时长`
   - 测试档位（每行一个）：`如：0小时 / 4小时 / 8小时`
   - 要观察的结果 + 测量方式 + 单位：`如：芽的平均长度，用直尺测量，毫米`
   - 工程/混合卡追加：证据如何返回下一版设计。
3. **③ 边界**：禁止方向、课程锚点、安全数值范围。
4. **④ 预览（核心新增）**：调用一个无副作用的预览 API（复用 `compileOneCase`，不落库），实时展示这张卡会生成什么——
   - 「学生会这样开场」：P1/P2/P4 各一条示例学生消息；
   - 「数据表长这样」：`deterministicRows` 生成的表格渲染出来；
   - 「导师会看到的可见事实」：中文化的 visibleFacts 摘要。
   校验错误（如"路线 2 缺少测量单位"）直接标注在对应步骤上。

**预览是消除"不知道怎么填"的关键**：`levels` 填得对不对，看一眼生成的数据表列名就知道；`studentOpening` 顺不顺，看一眼合成的学生消息就知道。

### 3.4 素材池（familyKey 隐形化）

- 分组展示改叫「同一课程项目」，组标题不再显示哈希键。
- 「合并/拆分 familyKey」按钮 + `window.prompt` 改为「归入同一项目」对话框：勾选素材 → 下拉选择现有项目或"新建项目"→ 确认。familyKey 由系统生成，用户永远不接触。
- 顶部那段 familyKey 解释横幅删除，替换为操作导向文案：「同一门课的课件、视频、任务单会归为一个项目，编译时按项目去重，不会重复出题。你的工作：① 确认授权 ② 补 20 字以上摘要 ③ 送编译。」
- 下拉选项去掉括注的枚举原文。

### 3.5 卡片列表

- 状态中文徽章：草稿（待审核）/ 已启用 / 已被新版替代 / 已拒绝；分区标题同步中文。
- 元信息行改为中文（版本 2 · 第 1 版 · 生命健康 · 工程设计 · 已生成 12 个案例）。
- 批量批准加确认对话框并报告成功/失败条数；拒绝理由改用对话框表单。

## 4. 案例批次页（run 中心 + 门禁 stepper）

- 页面主体改为纵向 stepper：**Smoke 6 → Calibration 12 → Trial 36 → Full 180 / EVAL 80**。当前层高亮；未解锁层置灰并显示解锁条件（「生成 180 正式集」补上与 Calibration/Trial 一致的 disabled 联动，禁用原因写在按钮下方）。
- 案例列表按 generation run 分组，每个 run 一张卡：进度摘要（待生成 x · 初审中 y · 待定稿 z · 已定稿 n）+ 门禁指标。指标键名全部中文化并标注达标线（如「直接确认率 92% / 需 ≥85%」），failures 用中文说明"没通过什么、差多少"。
- Trial 36 人工签署从 `window.prompt` 改为正式表单：逐项复盘清单（主题漂移 / 伪学生表达 各一栏结论）+ 签署人 + 提交确认。
- 「批量生成所有 READY A/B」加成本确认框（显示条数与预估调用次数）。
- 案例生成完成后显示交接横幅：「N 条已进入初审队列，等待标注员处理」。

## 5. 数据版本页 = 交付台

明确页面身份：**这是给外部算力平台的数据出口。**

- 三段式布局：① 选择已定稿数据创建版本（不合格条目置灰时注明原因：评测集专用 / 未获训练资格）→ ② 冻结（带确认对话框：冻结后不可修改）→ ③ 下载交付。
- 下载区每个文件写明用途，替代裸代号链接：
  - `training.json` — SFT 训练集，交给算力平台
  - `preference.json` — 偏好对（chosen/rejected），用于偏好训练
  - `manifest.json` — 校验清单（含 SHA-256，供双方核对）
  - `clean.json` — 完整标注数据（存档/审计用）
- 页尾引导：「外部训练完成后 → 前往模型档案登记训练结果」。
- 「从 FinalizedTutorTurn 创建并冻结版本」改为「从已定稿的导师回合创建数据版本」。

## 6. 模型档案页 = 迭代时间线

把 training-runs、evaluations、models、deployments 四页的登记功能整合为**每个模型版本一条纵向时间线**：

```
[当前生产模型卡：版本 · 灰度比例 · 运行时长]

模型版本 vX（外部平台训练）
 ├─ 数据来源     release-2026-07-xx（冻结哈希 …）
 ├─ 训练登记     日期 / 算力平台 / 父模型 / 超参备注   ← 外部训练完成后回填
 ├─ 评测回填     导入 transcript/verdict → 门禁结果（中文：通过/未通过 + 差在哪一项）
 └─ 部署状态     10% → 30% → 100% 进度条 + 观察窗口状态
                 观察指标表单：中文 label + 单位说明（如「结构解析失败率（0~1 小数）」）
                 回滚按钮（二次确认 + 后果说明）
```

- 评测导入表单补 transcript/verdict 的格式说明和示例下载；核验结果不再裸甩 JSON。
- 这个结构天然兼容"学校算力、服务与训练同机"的未来：届时训练登记从手工回填变成自动写入，页面结构不变。

## 7. 审核工作台修复（annotator / reviewer）

结构基本保留，修四个硬伤 + 减负：

1. **退回闭环**（伤害最大）：EDIT 领取时服务端返回 reviewer 退回理由与 annotator 上一稿（`bootstrap/service.ts` claim payload 目前只在 CONFIRM 返回 `firstReview`）；前端展示"被退回原因"横幅并恢复上次草稿。
2. **租约**：显示倒计时，剩 5 分钟黄色警告，提供「续租」；过期后立即锁定编辑并提示重新领取。
3. **错误人话化**：硬错误改用检查项自带的中文 `message`（现在只取英文 `code`）；warning 未完成提示改为"第 N 条信号"并高亮对应卡片（现在拼接数据库 cuid）；提交反馈状态码映射中文。
4. **初审页减负**：preference 对、submissionMode 自申报、REGRESSION/NEGATIVE 等治理选项收进「高级」折叠区；选择候选后手动编辑正文时自动把决定从 SELECT_A/B 切为 EDIT。
5. 两页顶部加 3~4 步操作指引条；案例元信息行（split/triggerType 等）中文化；对话历史（已返回未渲染）补渲染。

## 8. 横切规范（全站执行）

1. **统一术语映射层**：新建 `app/lib/dataLab/labels.ts`，收敛所有状态/枚举/warning code/指标键名 → 中文 label（+可选 help 文案）。`TutorLanguageEditor` 和 warningClosure 已有的映射迁入，全站消灭裸枚举。
2. `window.prompt`/`confirm` 全部替换为对话框组件（新建一个轻量 `Dialog`，全站复用）。
3. 不可逆操作（冻结、回滚、淘汰案例、归档）一律二次确认 + 后果说明。
4. 页面级权限统一：每个 page.tsx 顶部按角色 `redirect`，与 API 层一致（修复 annotator 手输 URL 可看 admin 数据的越权读取）。
5. 已冻结旧流程的写入 UI 拆除：batches 页移除导入表单、campaigns 页移除创建/启动控件，改为只读说明 +「历史数据」定位。
6. 清理 `styleMetadata.ts.tmp`。

## 9. 实施顺序

| 阶段 | 内容 | 特点 |
|---|---|---|
| P1 地基 | labels 术语层、文案三段式重写、prompt→Dialog、权限 guard、旧流程写入 UI 拆除、审核闭环两硬伤（退回理由/租约） | 不动结构，风险低，见效最快 |
| P2 动线 | 概览指挥台、导航重组、案例批次页 run-stepper、门禁按钮联动 | 中等改动 |

| P4 交付 | 数据版本交付台、模型档案时间线（合并四页） | 涉及页面合并 |
