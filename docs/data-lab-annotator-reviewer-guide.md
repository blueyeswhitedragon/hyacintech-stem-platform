# Hyacintech Data Lab 标注与复审使用教程

适用对象：受邀参与数据整理的标注者（annotator）和复审者（reviewer）。

本文以 2026-07-10 的本地实际界面为准。账号由管理员创建，后台账号不能自行注册。登录地址为 `http://localhost:3000/auth/login`；部署后请使用管理员提供的正式地址和初始密码。

## 一、先理解这项工作的目标

Data Lab 不是普通的文本润色工具。每条数据是一段学生与 AI 导师的 STEM 探究对话，标注目标是让导师回复同时满足：

- 不替学生完成思考，保留学生自主性；
- 不偏离学生原主题、原机制和当前探究阶段；
- 一次只推进一个核心问题，避免认知负担过高；
- 给出必要但不过度的安全提醒；
- 保持系统要求的结构化字段完整、合法；
- 语言自然、简洁，避免模板腔和空泛鼓励。

候选数据中的 `Gold candidate` 只是候选等级。只有经过人工标注和复审后，才可能成为 `Human Gold`。

## 二、标注者工作流程

### 1. 登录并领取任务

1. 使用管理员提供的标注者账号登录。
2. 进入 **我的标注**。
3. 页面会自动领取一条任务，并显示阶段、目标风格、场景名称和任务租约截止时间。

注意：打开 **我的标注** 就可能领取任务并开始租约。不要为了“随便看看”反复打开页面；领取后应尽量在租约时间内完成或先保存草稿。

管理员当前将每位标注者的同时活跃任务数限制为 1。提交后，页面会自动领取下一条任务，这是正常行为。

### 2. 识别只读和可编辑内容

- **学生 · 只读**：原始学生消息，不允许修改。
- **导师回复 · 可修订**：需要检查和修订的 AI 回复。
- 场景、阶段、学生消息、来源证据都必须保持不变。

每个导师回复可能包含：

- **对话内容**：学生实际看到的导师文本；
- **下一步动作**：继续输入、请求确认、信息提示或选择题；
- **提示**：每行一条；
- **选项**：仅在选择题中使用，每行一条；
- **当前阶段完成**：只有当前阶段确实达到完成条件时才能勾选；
- 阶段专属结构，例如阶段 1 的确认结构、阶段 2 的实验数据表、阶段 5 的报告结构。

### 3. 逐轮检查导师回复

建议按以下顺序检查：

1. 先读完整对话，弄清学生正在做什么。
2. 检查导师是否回应了学生刚说的内容，而不是套用通用模板。
3. 检查有没有替学生直接给出结论、方案或成品。
4. 检查是否一次提出过多问题。
5. 检查阶段是否越界，例如阶段 1 不应提前写完整实验方案。
6. 检查动作类型、提示、选项、完成状态和阶段结构是否互相一致。
7. 只修改确实需要修改的部分，避免无意义改写。

### 4. 阶段专项检查

#### P1 选题定向

- 保留学生最初兴趣和关键机制；
- 将宽泛兴趣收敛为可观察、可测量的问题；
- 不替学生直接决定完整研究题目；
- 阶段完成时确认结构必须完整，包括主题映射、研究问题、自变量和确认书；
- 阶段 1 不使用选择题式推进。

#### P2 方案设计

- 明确自变量、因变量和控制变量；
- 实验步骤应可执行、可记录、风险可控；
- 完成时必须提供合法数据表结构；
- 数据表应有 `notes` 备注列，最大行数为 200；
- 列名、数据类型和必填状态要能支持后续真实记录。

#### P3 过程执行

- 重点帮助学生按方案记录过程和数据；
- 安全提醒要针对真实风险，不要制造恐慌；
- 不虚构学生未实际获得的数据。

#### P4 数据分析

- 基于已有数据说话；
- 引导学生识别趋势、异常和证据限制；
- 不提前替学生下最终结论。

#### P5 报告成型

- 报告结构各部分应完整；
- 内容必须能追溯到前面阶段的真实主题、方案和数据；
- 不凭空补造实验结果。

#### P6 结果反思

- 引导学生反思证据质量、误差、局限和改进方向；
- 避免只给空泛表扬或重复总结。

### 5. 使用问题标签

发现问题时勾选对应标签：

| 标签 | 含义 |
|---|---|
| `student_agency_loss` | 导师代替学生思考、决定或完成任务 |
| `theme_drift` | 回复偏离学生原主题 |
| `proxy_drift` | 课堂代理任务改变了原问题的核心机制 |
| `cognitive_overload` | 一次给出太多要求、问题或信息 |
| `safety_missing` | 存在真实风险但缺少必要安全提醒 |
| `safety_overload` | 风险很低却给出过度、干扰性的安全警告 |
| `stage_violation` | 回复越过当前阶段边界 |
| `premature_conclusion` | 证据不足时提前下结论 |
| `formulaic_tone` | 模板腔、机械鼓励、重复铺垫 |
| `structure_invalid` | 动作类型或阶段专属结构不合法、不完整 |

标签应描述原回复或本次修订所解决的问题。不要为了“看起来认真”而随意多选。

### 6. 填写修改理由

修改理由应简短说明：

- 原回复的问题是什么；
- 你保留了什么；
- 你修复了什么；
- 为什么这样改更符合当前阶段和教学目标。

推荐写法：

> 删去偏抽象的类比和重复铺垫，使问题更简洁，并明确要求学生自己选择安全反应、自变量和可测结果。

不要只写“已优化”“修改措辞”或“更好”。

### 7. 无需修改的情况

如果所有导师回复都已经符合要求：

1. 不改动导师文本；
2. 勾选 **无需修改，原回复已符合要求**；
3. 在修改理由中简要说明检查结论；
4. 保存草稿或提交。

不要在已经改动内容后仍勾选“无需修改”。当前界面不会自动阻止这种矛盾，需要标注者自行避免。

### 8. 保存草稿和提交标注

- **保存草稿**：保存当前编辑结果，任务仍由你继续处理。页面会显示“草稿已保存”。
- **提交标注**：创建一个不可替代原始记录的新 revision，当前任务变为已提交，然后自动领取下一条任务。

提交前最后检查：

- 所有导师轮次都保留且可解析；
- 学生消息没有被修改；
- 动作类型和选项一致；
- 阶段完成状态正确；
- 阶段专属结构完整；
- 问题标签和修改理由与实际改动一致；
- “无需修改”状态没有与真实改动冲突。

## 三、复审者工作流程

复审者可以进入 **复审仲裁**，也可以承担普通标注任务。仲裁页面在决策前不会显示作者、目标风格、教师模型来源或候选等级。

### 1. 领取仲裁任务

进入 **复审仲裁** 后，系统自动领取一条待仲裁任务。页面显示：

- 阶段和场景；
- 原始学生消息；
- 匿名版本 A、B 等候选版本；
- 最终等级和仲裁理由。

### 2. 比较匿名版本

按以下顺序比较：

1. 是否忠实回应同一组学生消息；
2. 是否保留学生自主性和原主题机制；
3. 是否符合当前阶段；
4. 是否足够简洁、自然、可操作；
5. 结构化字段是否完整；
6. 安全提醒是否必要且适度。

不要根据版本顺序猜测作者或质量。A/B 标签每个仲裁任务都只是匿名展示。

### 3. 三种处理方式

- **接受所选**：先选择一个候选版本，再设置最终等级并填写理由。
  - `Human Gold`：质量高，可作为人工 Gold。
  - `Reviewed Silver`：基本可用，但不达到 Gold 标准。
- **退回**：将任务退回标注者继续修改；理由应明确指出可操作的问题。
- **拒绝**：该样本不应进入发布数据；说明拒绝原因。

当前网页界面只支持选择某个候选版本、退回或拒绝，不提供可视化的“合并两个版本”编辑器。

### 4. 仲裁理由要求

理由至少回答两个问题：

- 为什么这个版本优于另一个版本；
- 为什么它适合对应的最终等级。

推荐写法：

> 版本 A 更简洁，去除了偏抽象的类比，同时保留学生自主选择，并明确了安全反应、自变量和可测结果，因此接受为 Human Gold。

## 四、常见问题

### 页面一打开就出现任务

这是自动领取机制，不是系统错误。顶部租约时间表示该任务暂时由你处理。

### 概览显示“待办标注 0”，但工作台里有任务

概览数字只统计当前分配给你的 `PENDING/IN_PROGRESS` 任务；页面状态刷新、自动领取和提交后的短暂切换可能让数字与工作台显示不同。以 **我的标注** 当前工作台为准。

### 提交后为什么马上换成另一条任务

提交成功后系统会自动领取下一条，这是连续标注设计。若准备休息，不要开始编辑新任务；退出登录即可，新任务会保留到租约过期或下次继续。

### 什么时候应该退回而不是拒绝

- 有明确、可修复问题：退回；
- 样本主题、结构或证据基础不适合继续使用：拒绝。

### 发现系统错误怎么办

记录账号、场景名称、阶段、操作步骤和页面错误文字，交给管理员。不要通过反复提交来“试出结果”。

2026-07-10 已在 Chrome 中实际验证：

- 标注者自动领取和租约显示；
- 原始学生消息只读、导师回复可编辑；
- 阶段 1 和阶段 2 专属编辑结构；
- 问题标签、修改理由和“无需修改”；
- 保存草稿成功提示；

- 复审者看到匿名版本 A/B；
- 选择版本、填写理由并接受为 Human Gold；
- 仲裁完成后队列显示“当前没有待仲裁任务”。
