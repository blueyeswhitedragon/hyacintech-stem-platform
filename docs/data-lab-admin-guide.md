# Hyacintech Data Lab 管理员使用教程

适用对象：负责账号、数据批次、标注活动、版本发布、训练登记和评测归档的 Data Lab 管理员。

本文以 2026-07-10 的本地实际界面为准。管理员入口为 `http://localhost:3000/auth/login`，登录后自动进入 `/data-lab`。

## 一、管理员职责和完整流程

```text
创建后台账号
  → 导入不可变数据批次
  → 创建并启动标注活动
  → 标注者独立修订
  → 复审者匿名仲裁
  → 创建并冻结数据版本
  → 登记外部训练任务
  → 导入 transcript/verdict 双盲评测证据
```

管理员也能进入 **我的标注** 和 **复审仲裁**，但日常工作中建议保持角色分工，避免管理员身份参与普通标注后影响盲审纪律。

## 二、概览页

概览展示：

- 数据批次；
- 基线样本；
- 标注活动；
- 待办标注；
- 待仲裁；
- 数据版本；
- 训练任务；
- 评测记录。

管理员的“待办标注”包括活动中处于 `PENDING` 和 `IN_PROGRESS` 的任务。“待仲裁”包括 `PENDING` 和 `IN_REVIEW` 的仲裁任务。

建议每天开始和结束时记录一次概览数字，结合异常反馈判断是否存在任务长期占用、仲裁积压或活动配置错误。

## 三、后台账号管理

进入 **后台账号**。

### 创建账号

填写：

- 用户名：至少 3 个字符；
- 显示名称：建议使用真实可识别的工作名称；
- 初始密码：至少 8 个字符；
- 角色：标注者、复审者或管理员。

后台角色禁止自助注册。请通过安全的私下渠道发送初始密码，并要求参与者不要共用账号。

推荐命名：

- `annotator-姓名缩写`；
- `reviewer-姓名缩写`；
- 避免使用共享账号，例如 `team1`。

### 调整角色

列表中的角色下拉框会在选中后立即保存，没有额外确认步骤。操作前先核对用户名，避免误把标注者提升为管理员。

当前界面没有删除账号或重置密码功能。人员退出项目时，应先调整其角色或由技术管理员处理数据库和密码变更。

## 四、导入数据批次

进入 **数据批次**。

### 准备文件

- 必需：ShareGPT 风格数据集 JSON；
- 可选：Manifest JSON；
- 数据集顶层必须是数组；
- 单次上传上限为 10 MB；
- 每条记录至少包含合法 ID、场景、1–6 阶段和交替的 human/gpt 消息；
- gpt 消息必须是合法 `ChatResponse` JSON。

导入前建议先运行仓库内的数据校验脚本，不要把明显错误留给网页导入。

### 页面操作

1. 填写唯一的批次名称；
2. 选择来源类型：ShareGPT clean、人工修订或外部数据；
3. 选择数据集 JSON；
4. 如有，选择 Manifest；
5. 点击 **导入批次**；
6. 等待“导入完成：N 条”；
7. 核对样本数和自动检查汇总。

原始记录导入后保持不可变，后续修改创建 revision。批次名称和来源文件内容哈希应保持唯一，不要重复导入同一文件来解决配置问题。

历史 489 条 `dataset-base-v1` 已整体隔离，不得继续下发、冻结或登记训练。管理员页面仍可看到它们用于追溯，但新活动只会列出 `ACTIVE` 批次。新的 dataset schema v3 候选必须先通过自动契约校验，再由人工完成标注、工作量审核和仲裁；模型评估给出的 `gold_candidate` 不是人工 Gold。

## 五、创建和启动标注活动

进入 **标注活动**。

### 配置字段

- **活动名称**：使用批次、轮次和目的组成的唯一名称；
- **Gold 独立标注数**：允许 1–3，正式 Gold 建议 2；
- **Silver 双审比例**：0–100%；
- **样本上限**：0 表示全部，试运行建议 10–20；
- **数据批次**：选择活动的数据来源；
- **阶段**：P1–P6，可按本轮目标筛选；
- **候选等级**：Gold candidate、Silver；
- **风格配额权重**：0 表示禁用某风格，其他数值作为相对权重；
- 每位标注者同时活跃任务数当前固定为 1。

五种风格为：苏格拉底简洁型、温和陪伴型、工程导师型、证据分析型、课堂教练型。

### 推荐流程

1. 先创建 10–20 条的小型校准活动；
2. 点击 **创建活动**，生成 `DRAFT` 草稿；
3. 在列表中核对筛选范围和预期任务量；
4. 点击草稿活动对应的 **启动活动**；
5. 状态变为 `ACTIVE` 后，标注者才能领取任务。

活动启动后会根据 Gold/Silver 协议创建任务。Gold 通常产生多份独立标注；Silver 默认单审，其中一部分按“双审比例”进入双人流程。

## 六、监控标注和仲裁

### 标注队列

任务常见状态：

- `PENDING`：尚未领取；
- `IN_PROGRESS`：已领取并在租约内；
- `SUBMITTED`：已提交 revision；
- `RETURNED`：复审退回，等待原标注者继续修改。

打开 **我的标注** 会自动领取任务，提交后也会自动领取下一条。因此外部标注人员开始工作前，应明确这一行为。

### 仲裁队列

Gold 的多份标注完成后，系统生成匿名 Review Case。复审者看不到作者、风格、来源模型或候选等级，只比较版本内容。

管理员应关注：

- 长期停留在 `IN_PROGRESS` 的任务；
- 退回次数异常高的阶段或标注者；
- `structure_invalid`、`stage_violation` 等高风险标签；
- Gold 候选是否完成独立标注和最终仲裁。

## 七、创建和冻结数据版本

进入 **数据版本**。

### 创建草稿

1. 填写唯一版本号，例如 `stem-sft-v1`；
2. 选择对应标注活动；
3. 设置 Gold 和 Silver 权重；
4. 点击 **创建版本草稿**。

默认配方：

- Human Gold 权重 1.5；
- Reviewed Silver 权重 1.0；
- 同时包含 Human Gold 和 Reviewed Silver。

### 冻结前检查

- 活动已经产生至少一个人工审定样本；
- Gold 样本完成匿名仲裁；
- Silver 样本满足单审或抽样双审要求；
- 版本号、权重和活动选择正确；
- `data/releases` 已备份或 Git 工作区状态已确认。

### 冻结发布

点击 **冻结发布** 后，系统会：

- 固化 ReleaseItem；
- 生成 all/clean、gold、silver 和 manifest 四类文件；
- 记录 SHA-256；
- 写入 `data/releases`；
- 状态变为 `FROZEN`。

只有 `DRAFT` 可以冻结，冻结后不可修改。不要用正式版本号做界面试验。

冻结后列表提供 `clean`、`gold`、`silver`、`manifest` 下载链接。Manifest 包含配方、阶段分布、样本和 revision 证据链。

## 八、登记外部训练任务

进入 **训练登记**。

Data Lab 只记录训练任务，不自动向主办方平台提交任务，也不保存平台密钥。

填写：

- 任务名称；
- 已冻结的数据版本；
- 基础模型；
- 主办方任务 ID；
- 模型标签；
- 状态：DRAFT、SUBMITTED、RUNNING、SUCCEEDED、FAILED；
- 训练参数 JSON；
- 备注。

训练参数必须是合法 JSON。登记前将参数从主办方页面完整复制并二次核对，尤其是学习率、epoch、batch size 和随机种子。

## 九、导入双盲评测

进入 **双盲评测**。

1. 用 CLI 完成模型采集和裁判；
2. 为本轮评测填写唯一名称；
3. 一次选择一个或多个 transcript/verdict JSON；
4. 点击 **导入评测**；
5. 核对模型 A/B、评测范围、产物数、胜负和平局统计。

网页负责证据归档和摘要展示，不替代 CLI 的采集与裁判流程。

## 十、给外部数据整理人员的推荐运营方式

### 上线前

1. 每人创建独立账号；
2. 发放《标注与复审使用教程》；
3. 用 10–20 条校准活动培训；
4. 让所有人独立完成同一小组样本；
5. 复盘分歧，统一 Gold 标准和问题标签口径；
6. 校准通过后再启动正式活动。

### 进行中

- 每天查看待办和待仲裁数字；
- 抽查“无需修改”比例和修改理由质量；
- 对同一标签频繁出现的样本做专题复盘；
- 不在活动中途随意修改人员角色；
- 不让标注者看到另一个独立版本；
- 不用聊天软件直接传输完整数据集文件，统一在 Data Lab 中工作。

### 发布前

- 先确认工作区和数据库备份；
- 抽查各阶段与各风格样本；
- 核对 Gold/Silver 数量和权重；
- 冻结后保存 manifest 和哈希；
- 用冻结版本登记训练任务，避免训练数据与记录脱节。

## 十一、当前界面的重要限制

- 没有账号删除和密码重置页面；
- 调整角色会立即保存，没有二次确认；
- 活动、批次和冻结版本没有删除入口；
- 冻结版本不可修改；
- 复审网页不提供候选版本合并编辑器；
- 标注者可同时勾选“无需修改”并编辑内容，系统不会在界面上阻止，需要流程规范约束；
- 任务提交后自动领取下一条，可能产生暂时无人继续处理的 `IN_PROGRESS` 任务；
- 数据库为本机 SQLite，正式多人使用前需要制定备份、并发和部署策略。

## 十二、本轮实测记录

2026-07-10 已在 Chrome 中实际完成：

- 管理员概览、数据批次、活动、账号、版本、训练和评测页面检查；
- 两位标注者对同一 Gold 样本分别提交“无需修改”和真实修订版本；
- 草稿保存与提交后自动领取；
- 自动生成匿名仲裁任务；
- 复审者比较版本 A/B，选择修订版并接受为 Human Gold；
- 仲裁队列完成后归零；
- 数据库审计日志记录领取、提交、领取仲裁和仲裁决策。

为保护试运行数据，本轮没有执行新的批次导入、账号创建、活动创建、版本冻结、训练登记或评测导入。
